The `SHOW TRACE FOR SESSION` [statement](/docs/v23.2/sql-statements) returns details about how CockroachDB executed a statement or series of statements recorded during a session. These details include messages and timing information from all nodes involved in the execution, providing visibility into the actions taken by CockroachDB across all of its software layers.

You can use `SHOW TRACE FOR SESSION` to debug why a query is not performing as expected, to add more information to bug reports, or to generally learn more about how CockroachDB works.

A [statement diagnostics bundle](/docs/v23.2/ui-statements-page#diagnostics) contains statement traces in plaintext, JSON, and [Jaeger-compatible](/docs/v23.2/query-behavior-troubleshooting#visualize-statement-traces-in-jaeger) format.

## Usage overview

`SHOW TRACE FOR SESSION` returns [statement traces](#trace-description) for the most recently executed statements.

To start recording statement traces during a session, enable the `tracing` session variable via [`SET tracing = on;`](/docs/v23.2/set-vars#set-tracing). To stop recording statement traces during a session, disable the `tracing` session variable via [`SET tracing = off;`](/docs/v23.2/set-vars#set-tracing).

Recording statement traces during a session does not effect the logical execution of the statements. This means that errors encountered by statements during a recording are returned to clients. CockroachDB will [automatically retry](/docs/v23.2/transactions#automatic-retries) individual statements (considered implicit transactions) and multi-statement transactions sent as a single batch when [retry errors](/docs/v23.2/transactions#error-handling) are encountered due to [contention](/docs/v23.2/performance-best-practices-overview#transaction-contention). Also, clients will receive retry errors required to handle [client-side transaction retries](/docs/v23.2/transaction-retry-error-reference#client-side-retry-handling). As a result, traces of all transaction retries will be captured during a recording.

## Required privileges

For `SHOW TRACE FOR SESSION`, no privileges are required.

## Trace description

CockroachDB uses [OpenTelemetry](https://opentelemetry.io/docs/concepts/data-sources) libraries for tracing, which also means that it can be easily integrated with OpenTelemetry-compatible [trace collectors](/docs/v23.2/query-behavior-troubleshooting#configure-cockroachdb-to-send-traces-to-a-third-party-trace-collector). CockroachDB traces map to OpenTelemetry trace and span concepts as follows:

<table><thead><tr><th>Concept</th><th>Description</th></tr></thead><tbody><tr><td><strong>trace</strong></td><td>Information about the sub-operations performed as part of a high-level operation (a query or a transaction). This information is internally represented as a tree of "spans", with a special "root span" representing a whole SQL transaction in the case of <code>SHOW TRACE FOR SESSION</code>.</td></tr><tr><td><strong>span</strong></td><td>A named, timed operation that describes a contiguous segment of work in a trace. Each span links to "child spans", representing sub-operations; their children would be sub-sub-operations of the grandparent span, etc.<br/><br/>Different spans can represent (sub-)operations that executed either sequentially or in parallel with respect to each other. (This possibly-parallel nature of execution is one of the important things that a trace is supposed to describe.) \The operations described by a trace may be <em>distributed</em>, that is, different spans may describe operations executed by different nodes.</td></tr><tr><td><strong>message</strong></td><td>A string with timing information. Each span can contain a list of these. They are produced by CockroachDB's logging infrastructure and are the same messages that can be found in node <a href="/docs/v23.2/logging-overview">log files</a> except that a trace contains message across all severity levels, whereas log files, by default, do not. Thus, a trace is much more verbose than logs but only contains messages produced in the context of one particular traced operation.</td></tr></tbody></table>

Consider a visualization of a trace for one statement as [visualized by Jaeger](/docs/v23.2/query-behavior-troubleshooting#visualize-statement-traces-in-jaeger). The image shows spans and log messages. You can see names of operations and sub-operations, along with parent-child relationships and timing information, and it's easy to see which operations are executed in parallel.

## Response

<Note>
The format of the `SHOW TRACE FOR SESSION` response may change in future versions.
</Note>

CockroachDB outputs traces in linear tabular format. Each result row represents either a span start (identified by the `=== SPAN START: <operation> ===` message) or a log message from a span. Rows are generally listed in their timestamp order (i.e., the order in which the events they represent occurred) with the exception that messages from child spans are interleaved in the parent span according to their timing. Messages from sibling spans, however, are not interleaved with respect to one another.

The following diagram shows the order in which messages from different spans would be interleaved in an example trace. Each box is a span; inner-boxes are child spans. The numbers indicate the order in which the log messages would appear in the virtual table.

```text
 +-----------------------+
 |           1           |
 | +-------------------+ |
 | |         2         | |
 | |  +----+           | |
 | |  |    | +----+    | |
 | |  | 3  | | 4  |    | |
 | |  |    | |    |  5 | |
 | |  |    | |    | ++ | |
 | |  +----+ |    |    | |
 | |         +----+    | |
 | |          6        | |
 | +-------------------+ |
 |            7          |
 +-----------------------+
```

Each row contains the following columns:

<table><thead><tr><th>Column</th><th>Type</th><th>Description</th></tr></thead><tbody><tr><td><code>timestamp</code></td><td><code>TIMESTAMPTZ</code></td><td>The absolute time when the message occurred.</td></tr><tr><td><code>age</code></td><td><code>INTERVAL</code></td><td>The age of the message relative to the beginning of the trace (i.e., the beginning of the recording in the case of <code>SHOW TRACE FOR SESSION</code>).</td></tr><tr><td><code>message</code></td><td><code>STRING</code></td><td>The log message.</td></tr><tr><td><code>tag</code></td><td><code>STRING</code></td><td>Meta-information about the message's context. This is the same information that appears in the beginning of log file messages in between square brackets (e.g, <code>&#91;client=&#91;::1&#93;:49985,user=root,n1&#93;</code>).</td></tr><tr><td><code>location</code></td><td><code>STRING</code></td><td>The <code>file:line</code> location of the line of code that produced the message. Only some of the messages have this field set; it depends on specifically how the message was logged. The <code>--vmodule</code> flag passed to the node producing the message also affects what rows get this field populated. Generally, if <code>--vmodule=&lt;file&gt;=&lt;level&gt;</code> is specified, messages produced by that file will have the field populated.</td></tr><tr><td><code>operation</code></td><td><code>STRING</code></td><td>The name of the operation (or sub-operation) on whose behalf the message was logged.</td></tr><tr><td><code>span</code></td><td><code>INT</code></td><td>The index of the span within the virtual list of all spans if they were ordered by the span's start time.</td></tr></tbody></table>

If you specify the `COMPACT` parameter, only the `age`, `message`, `tag`, and `operation` columns are returned. In addition, the value of the `location` column is prepended to `message`.

```sql
> SET tracing = on;
```

```sql
SET TRACING
```

> SHOW TRACE FOR SESSION;

### Trace conflicting transactions

This example uses two terminals concurrently to generate conflicting transactions.

1. In terminal 1, create a table:

    ```sql
    > CREATE TABLE t (k INT);
    ```
2. In terminal 1, open a transaction and perform a write without closing the transaction:

    ```sql
    > BEGIN;
    ```

    ```sql
    > INSERT INTO t VALUES (1);
    ```

    Press enter one more time to send these statements to the server.
3. In terminal 2, turn tracing on:

    ```sql
    > SET tracing = on;
    ```
4. In terminal 2, execute a conflicting read:

    ```sql
    > SELECT * FROM t;
    ```

    You'll see that this statement is blocked until the transaction in terminal 1 finishes.
5. In terminal 1, finish the transaction:

    ```sql
    > COMMIT;
    ```
6. In terminal 2, you'll see the completed read:

    ```text
      k
    +---+
      1
    (1 row)
    ```
7. In terminal 2, stop tracing and then view the completed trace:

    ```sql
    > SET tracing = off;
    ```

    ```sql
    > SHOW TRACE FOR SESSION;
    ```
