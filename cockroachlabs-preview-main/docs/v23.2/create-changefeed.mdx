The `CREATE CHANGEFEED` [statement](/docs/v23.2/sql-statements) creates a new Enterprise changefeed, which targets an allowlist of tables called "watched rows". Every change to a watched row is emitted as a record in a configurable format (`JSON` or Avro) to a configurable sink ([Kafka](https://kafka.apache.org/), [Google Cloud Pub/Sub](https://cloud.google.com/pubsub), a [cloud storage sink](/docs/v23.2/changefeed-sinks#cloud-storage-sink), or a [webhook sink](/docs/v23.2/changefeed-sinks#webhook-sink)). You can [create](#examples), [pause](#pause-a-changefeed), [resume](#resume-a-paused-changefeed), [alter](/docs/v23.2/alter-changefeed), or [cancel](#cancel-a-changefeed) an Enterprise changefeed.

To get started with changefeeds, refer to the [Create and Configure Changefeeds](/docs/v23.2/create-and-configure-changefeeds) page for important usage considerations. For detail on how changefeeds emit messages, refer to the [Changefeed Messages](/docs/v23.2/changefeed-messages) page.

The [examples](#examples) on this page provide the foundational syntax of the `CREATE CHANGEFEED` statement. For examples on more specific use cases with changefeeds, refer to the following pages:

- [Change Data Capture Queries](/docs/v23.2/cdc-queries)
- [Changefeeds on Tables with Column Families](/docs/v23.2/changefeeds-on-tables-with-column-families)
- [Export Data with Changefeeds](/docs/v23.2/export-data-with-changefeeds)

Cockroach Labs recommends monitoring your changefeeds to track [retryable errors](/docs/v23.2/monitor-and-debug-changefeeds#changefeed-retry-errors) and [protected timestamp](/docs/v23.2/architecture/storage-layer#protected-timestamps) usage. Refer to the [Monitor and Debug Changefeeds](/docs/v23.2/monitor-and-debug-changefeeds) page for more information.

## Required privileges

You can [grant](/docs/v23.2/grant#grant-privileges-on-specific-tables-in-a-database) a user the `CHANGEFEED` privilege to allow them to create changefeeds on a specific table:

```sql
GRANT CHANGEFEED ON TABLE example_table TO user;
```

When you grant a user the `CHANGEFEED` privilege on a set of tables, they can:

- Create changefeeds on the target tables even if the user does **not** have the [`CONTROLCHANGEFEED` role option](/docs/v23.2/alter-role#role-options) or the `SELECT` privilege on the tables.
- Manage the changefeed jobs running on the tables using the [`SHOW CHANGEFEED JOB`](/docs/v23.2/show-jobs#show-changefeed-jobs), [`PAUSE JOB`](/docs/v23.2/pause-job), [`RESUME JOB`](/docs/v23.2/resume-job), and [`CANCEL JOB`](/docs/v23.2/cancel-job) commands.

These users will be able to create changefeeds, but they will not be able to run a `SELECT` query on that data directly. However, they could still read this data indirectly if they have read access to the [sink](/docs/v23.2/changefeed-sinks).

To restrict a user's access to changefeed data and sink credentials, enable the `changefeed.permissions.require_external_connection_sink.enabled` cluster setting. When you enable this setting, users with the [`CHANGEFEED` privilege](/docs/v23.2/create-changefeed#required-privileges) on a set of tables can only create changefeeds into [external connections](/docs/v23.2/create-external-connection).

You can add `CHANGEFEED` to the user or role's [default privileges](/docs/v23.2/security-reference/authorization#default-privileges) with [`ALTER DEFAULT PRIVILEGES`](/docs/v23.2/alter-default-privileges#grant-default-privileges-to-a-specific-role):

```sql
ALTER DEFAULT PRIVILEGES GRANT CHANGEFEED ON TABLES TO user;
```

### Sink URI

This section provides example URIs for each of the sinks that CockroachDB changefeeds support. For more comprehensive detail of using and configuring each sink, refer to the [Changefeed Sinks](/docs/v23.2/changefeed-sinks) page.

```text
'{scheme}://{host}:{port}?{query_parameters}'
```

<table><thead><tr><th>URI Component</th><th>Description</th></tr></thead><tbody><tr><td><code>scheme</code></td><td>The type of sink: <a href="#kafka"><code>kafka</code></a>, <a href="#google-cloud-pub-sub"><code>gcpubsub</code></a>, any <a href="#cloud-storage">cloud storage sink</a>, or <a href="#webhook">webhook sink</a>.</td></tr><tr><td><code>host</code></td><td>The sink's hostname or IP address.</td></tr><tr><td><code>port</code></td><td>The sink's port.</td></tr><tr><td><code>query&#95;parameters</code></td><td>The sink's <a href="#query-parameters">query parameters</a>.</td></tr></tbody></table>

You can create an external connection to represent a changefeed sink URI. This allows you to specify the external connection's name in statements rather than the provider-specific URI. For detail on using external connections, see the [`CREATE EXTERNAL CONNECTION`](/docs/v23.2/create-external-connection) page.

#### Kafka

Example of a [Kafka sink](/docs/v23.2/changefeed-sinks#kafka) URI:

```text
'kafka://broker.address.com:9092?topic_prefix=bar_&tls_enabled=true&ca_cert=LS0tLS1CRUdJTiBDRVJUSUZ&sasl_enabled=true&sasl_user={sasl user}&sasl_password={url-encoded password}&sasl_mechanism=SCRAM-SHA-256'
```

<Note>
[VPC Peering](/docs/cockroachcloud/network-authorization#vpc-peering) and [AWS PrivateLink](/docs/cockroachcloud/network-authorization#aws-privatelink) in CockroachDB Advanced clusters do **not** support connecting to a [Kafka](/docs/v23.2/changefeed-sinks#kafka) sink's internal IP addresses for [changefeeds](/docs/v23.2/change-data-capture-overview). To connect to a Kafka sink from CockroachDB Advanced, it is necessary to expose the Kafka cluster's external IP address and open ports with firewall rules to allow access from a CockroachDB Advanced cluster.
</Note>

#### Confluent Cloud

<InlineImage alt="Megaphone" src="/images/common/icon-megaphone.png" /> New in v23.2: Example of a [Confluent Cloud sink](/docs/v23.2/changefeed-sinks#confluent-cloud) URI:

```text
'confluent-cloud://pkc-lzvrd.us-west4.gcp.confluent.cloud:9092?api_key={API key}&api_secret={url-encoded API secret}'
```

#### Google Cloud Pub/Sub

<Note>
**This feature is in [preview](/docs/v23.2/cockroachdb-feature-availability)** and subject to change. To share feedback and/or issues, contact [Support](https://support.cockroachlabs.com/hc/en-us).
</Note>

Example of a Google Cloud Pub/Sub sink URI:

```text
'gcpubsub://{project name}?region={region}&topic_name={topic name}&AUTH=specified&CREDENTIALS={base64-encoded key}'
```

In CockroachDB v23.2 and later, the `changefeed.new_pubsub_sink_enabled` cluster setting is enabled by default, which provides improved throughput. For details on the changes to the message format, refer to [Pub/Sub sink messages](/docs/v23.2/changefeed-sinks#pub-sub-sink-messages).

[Use Cloud Storage for Bulk Operations](/docs/v23.2/cloud-storage-authentication) explains the requirements for the authentication parameter with `specified` or `implicit`. Refer to [Changefeed Sinks](/docs/v23.2/changefeed-sinks#google-cloud-pub-sub) for further consideration.

## Files

The files emitted to a sink use the following naming conventions:

- [General file format](#general-file-format)
- [Resolved file format](#resolved-file-format)

<Note>
The timestamp format is `YYYYMMDDHHMMSSNNNNNNNNNLLLLLLLLLL`.
</Note>

### General file format

```text
/[date]/[timestamp]-[uniquer]-[topic]-[schema-id]
```

For example:

```text
/2020-04-02/202004022058072107140000000000000-56087568dba1e6b8-1-72-00000000-test_table-1.ndjson
```

When emitting changefeed messages to a [cloud storage sink](/docs/v23.2/changefeed-sinks#cloud-storage-sink), you can specify a partition format for your files using the [`partition_format`](#partition-format) query parameter. This will result in the following file path formats:

- `daily`: This is the default option and will follow the same pattern as the previous general file format.
- `hourly`: This will partition into an hourly directory as the changefeed emits messages, like the following:

    ```text
    /2020-04-02/20/202004022058072107140000000000000-56087568dba1e6b8-1-72-00000000-test_table-1.ndjson
    ```
- `flat`: This will result in no file partitioning. The cloud storage path you specify when creating a changefeed will store all of the message files with no additional directories created.

### Resolved file format

```text
/[date]/[timestamp].RESOLVED
```

For example:

```text
/2020-04-04/202004042351304139680000000000000.RESOLVED
```

## Examples

Before running any of the examples in this section it is necessary to [enable the `kv.rangefeed.enabled` cluster setting](/docs/v23.2/create-and-configure-changefeeds#enable-rangefeeds). If you are working on a CockroachDB Standard or Basic cluster, this cluster setting is enabled by default.

The following examples show the syntax for managing changefeeds and starting changefeeds with different use cases and features. The [Options](#options) table on this page provides a list of all the available options. For information on sink-specific query parameters and configurations, refer to the [Changefeed Sinks](/docs/v23.2/changefeed-sinks) page.

You can create an external connection to represent a changefeed sink URI. This allows you to specify the external connection's name in statements rather than the provider-specific URI. For detail on using external connections, see the [`CREATE EXTERNAL CONNECTION`](/docs/v23.2/create-external-connection) page.

<Note>
We recommend limiting the number of changefeeds per cluster to 80.
</Note>

### Create a changefeed connected to a sink

You can connect a changefeed to the following sinks:

- Kafka
- Cloud storage / HTTP
- Google Cloud Pub/Sub
- Webhook

```sql
CREATE CHANGEFEED FOR TABLE table_name, table_name2, table_name3
  INTO 'scheme://host:port'
  WITH updated, resolved;
```

For guidance on the sink URI, refer to:

- The [Changefeed Sinks](/docs/v23.2/changefeed-sinks) page for general detail on query parameters and sink configuration.
- The [Cloud Storage Authentication](/docs/v23.2/cloud-storage-authentication) page for instructions on setting up each supported cloud storage authentication.

### Create a changefeed that filters and transforms change data

[CDC queries](/docs/v23.2/cdc-queries) can filter and transform change data before emitting it to a sink or [a SQL client](#create-a-sinkless-changefeed).

You can adapt a changefeed with CDC queries by including `SELECT` and `WHERE` clauses in your `CREATE` statement:

```sql
CREATE CHANGEFEED INTO 'scheme://host:port'
  WITH updated, resolved
  AS SELECT owner_id, status
  FROM vehicles
  WHERE status = 'lost';
```

CDC queries can only run on a single table per changefeed and require an Enterprise license.

### Create a sinkless changefeed

You can create a changefeed that will send messages to the SQL client rather than a sink:

```sql
CREATE CHANGEFEED FOR TABLE table_name, table_name2, table_name3
  WITH updated, resolved;
```

Sinkless changefeeds do not require an Enterprise license; however, a sinkless changefeed with CDC queries **does** require an Enterprise license.

To create a sinkless changefeed using CDC queries:

```sql
CREATE CHANGEFEED WITH updated, resolved
  AS SELECT owner_id, status
  FROM vehicles
  WHERE status = 'lost';
```

### Use an external connection to specify a changefeed sink

[External connections](/docs/v23.2/create-external-connection) provide a way to define a name for a sink, which you can use instead of the provider-specific URI.

To restrict a user's access to changefeed data and sink credentials, enable the `changefeed.permissions.require_external_connection_sink.enabled` cluster setting. When you enable this setting, users with the [`CHANGEFEED` privilege](/docs/v23.2/create-changefeed#required-privileges) on a set of tables can only create changefeeds into [external connections](/docs/v23.2/create-external-connection).

External connections support all changefeed sinks.

```sql
CREATE EXTERNAL CONNECTION kafka_sink
  AS 'kafka://broker.address.com:9092?topic_prefix=bar_&tls_enabled=true&ca_cert={certificate}&sasl_enabled=true&sasl_user={sasl user}&sasl_password={url-encoded password}&sasl_mechanism=SCRAM-SHA-256';
```

In the changefeed statement, you specify the external connection name:

```sql
CREATE CHANGEFEED FOR TABLE table_name INTO 'external://kafka_sink'
  WITH resolved;
```

### Disallow schema changes on tables to improve changefeed performance

<InlineImage alt="Megaphone" src="/images/common/icon-megaphone.png" /> New in v23.2.1: Use the `schema_locked` [storage parameter](/docs/v23.2/with-storage-parameter) to disallow [schema changes](/docs/v23.2/online-schema-changes) on a watched table, which helps to decrease the latency between a write committing to a table and it emitting to the [changefeed's sink](/docs/v23.2/changefeed-sinks). You can lock the table before creating a changefeed or while a changefeed is running, which will enable the performance improvement for changefeeds watching the particular table.

Enable `schema_locked` on the watched table with the [`ALTER TABLE`](/docs/v23.2/alter-table) statement:

```sql
ALTER TABLE watched_table SET (schema_locked = true);
```

While `schema_locked` is enabled on a table, attempted schema changes on the table will be rejected and an error returned. If you need to run a schema change on the locked table, unlock the table with `schema_locked = false`, complete the schema change, and then lock the table again with `schema_locked = true`. The changefeed will run as normal while `schema_locked = false`, but it will not benefit from the performance optimization.

```sql
ALTER TABLE watched_table SET (schema_locked = false);
```

### Manage a changefeed

For Enterprise changefeeds, use [`SHOW CHANGEFEED JOBS`](/docs/v23.2/show-jobs) to check the status of your changefeed jobs:

```sql
SHOW CHANGEFEED JOBS;
```

Use the following SQL statements to pause, resume, or cancel a changefeed.

#### Pause a changefeed

```sql
PAUSE JOB job_id;
```

For more information, see [`PAUSE JOB`](/docs/v23.2/pause-job).

#### Resume a paused changefeed

```sql
RESUME JOB job_id;
```

For more information, see [`RESUME JOB`](/docs/v23.2/resume-job).

#### Cancel a changefeed

```sql
CANCEL JOB job_id;
```

For more information, see [`CANCEL JOB`](/docs/v23.2/cancel-job).

#### Modify a changefeed

To modify an Enterprise changefeed, [pause](/docs/v23.2/create-and-configure-changefeeds#pause) the job and then use:

```sql
ALTER CHANGEFEED job_id {ADD table DROP table SET option UNSET option};
```

You can add new table targets, remove them, set new [changefeed options](/docs/v23.2/create-changefeed#options), and unset them.

For more information, see [`ALTER CHANGEFEED`](/docs/v23.2/alter-changefeed).

#### Configuring all changefeeds

It is useful to be able to pause all running changefeeds during troubleshooting, testing, or when a decrease in CPU load is needed.

To pause all running changefeeds:

```sql
PAUSE JOBS (WITH x AS (SHOW CHANGEFEED JOBS) SELECT job_id FROM x WHERE status = ('running'));
```

This will change the status for each of the running changefeeds to `paused`, which can be verified with [`SHOW CHANGEFEED JOBS`](/docs/v23.2/show-jobs#show-changefeed-jobs).

To resume all running changefeeds:

```sql
RESUME JOBS (WITH x AS (SHOW CHANGEFEED JOBS) SELECT job_id FROM x WHERE status = ('paused'));
```

This will resume the changefeeds and update the status for each of the changefeeds to `running`.

### Start a new changefeed where another ended

In some situations, you may want to start a changefeed where a previously running changefeed ended. For example, a changefeed could encounter an error it cannot recover from, such as when a [`TRUNCATE` is performed](/docs/v23.2/create-and-configure-changefeeds#known-limitations), and you need to restart the changefeed.

1. Use [`SHOW CHANGEFEED JOB`](/docs/v23.2/show-jobs#show-changefeed-jobs) to find the [high-water timestamp](/docs/v23.2/monitor-and-debug-changefeeds#monitor-a-changefeed) for the ended changefeed:

    ```sql
    SHOW CHANGEFEED JOB {job_id};
    ```

    ```text
            job_id       | ... |      high_water_timestamp      | ...
    +--------------------+ ... +--------------------------------+ ...
      383870400694353921 | ... | 1537279405671006870.0000000000 | ...
    (1 row)
    ```

    <Note>
    If a changefeed has failed, you must restart the changefeed from a timestamp **after** the event that caused the failure.
    </Note>
2. Use the `high_water_timestamp` to start the new changefeed:

    ```sql
    CREATE CHANGEFEED FOR TABLE table_name, table_name2, table_name3
      INTO 'scheme//host:port'
      WITH cursor = '<high_water_timestamp>';
    ```

When you use the `cursor` option to start a changefeed, it will not perform an [initial scan](#initial-scan).

### Create a changefeed with an S3 storage class

To associate the changefeed message files with a [specific storage class](/docs/v23.2/use-cloud-storage#amazon-s3-storage-classes) in your Amazon S3 bucket, use the `S3_STORAGE_CLASS` parameter with the class. For example, the following S3 connection URI specifies the `INTELLIGENT_TIERING` storage class:

```sql
CREATE CHANGEFEED FOR TABLE table_name
  INTO 's3://{BUCKET NAME}?AWS_ACCESS_KEY_ID={KEY ID}&AWS_SECRET_ACCESS_KEY={SECRET ACCESS KEY}&S3_STORAGE_CLASS=INTELLIGENT_TIERING'
  WITH resolved;
```

Use the parameter to set one of these [storage classes](https://docs.aws.amazon.com/AmazonS3/latest/API/API_PutObject#AmazonS3-PutObject-request-header-StorageClass) listed in Amazon's documentation. For more general usage information, see Amazon's [Using Amazon S3 storage classes](https://docs.aws.amazon.com/AmazonS3/latest/userguide/storage-class-intro) documentation.

### Define a key to determine the changefeed sink partition

With the [`key_column`](#key-column) option, you can define the key used in message metadata that determines the partition for the changefeed message at your [downstream sink](/docs/v23.2/changefeed-sinks). This option overrides the default [primary key](/docs/v23.2/primary-key):

```sql
CREATE CHANGEFEED FOR TABLE table_name
  INTO 'external://kafka-sink'
  WITH key_column='partition_column', unordered;
```

`key_column` does not preserve ordering of messages from CockroachDB to the downstream sink, therefore you **must** include the [`unordered`](#unordered) option. It does not affect per-key [ordering guarantees](/docs/v23.2/changefeed-messages#ordering-and-delivery-guarantees) or the output of [`key_in_value`](#key-in-value).
