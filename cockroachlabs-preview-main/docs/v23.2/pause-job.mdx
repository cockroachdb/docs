The `PAUSE JOB` [statement](/docs/v23.2/sql-statements) lets you pause the following types of jobs:

- [`IMPORT`](/docs/v23.2/import) jobs
- [`BACKUP`](/docs/v23.2/backup) and [`RESTORE`](/docs/v23.2/restore) jobs
- [User-created table statistics](/docs/v23.2/create-statistics) jobs
- [Automatic table statistics](/docs/v23.2/cost-based-optimizer#table-statistics) jobs
- [Changefeeds](/docs/v23.2/create-changefeed)
- [Schema change](/docs/v23.2/online-schema-changes) jobs
- [Scheduled backup](/docs/v23.2/manage-a-backup-schedule) jobs

After pausing jobs, you can resume them with [`RESUME JOB`](/docs/v23.2/resume-job).

<Note>
If a schema change job is paused, any jobs waiting on that schema change will stop waiting and return an error.
</Note>

## Required privileges

To pause a job, the user must be a member of the `admin` role or must have the [`CONTROLJOB`](/docs/v23.2/create-user#create-a-user-that-can-pause-resume-and-cancel-non-admin-jobs) [role option](/docs/v23.2/security-reference/authorization#role-options) set. Non-admin users cannot pause admin users' jobs.

For changefeeds, users with the [`CHANGEFEED`](/docs/v23.2/create-changefeed#required-privileges) privilege on a set of tables can pause changefeed jobs running on those tables.

## Monitoring paused jobs

We recommend monitoring paused jobs. Jobs that are paused for a long period of time can start to affect the cluster in the following ways:

- A paused [backup](/docs/v23.2/backup), [restore](/docs/v23.2/restore), or index backfill job ( [schema change](/docs/v23.2/online-schema-changes) ) will continue to hold a [protected timestamp](/docs/v23.2/architecture/storage-layer#protected-timestamps) record on the data the job is operating on. This could result in data accumulation as the older versions of the keys cannot be [garbage collected](/docs/v23.2/architecture/storage-layer#garbage-collection). In turn, this may cause increased disk usage and degraded performance for some workloads. See [Protected timestamps and scheduled backups](/docs/v23.2/create-schedule-for-backup#protected-timestamps-and-scheduled-backups) for more detail.
- A paused [changefeed](/docs/v23.2/create-changefeed) job, if [`protect_data_from_gc_on_pause`](/docs/v23.2/create-changefeed#protect-pause) is set, will also hold a protected timestamp record on the data the job is operating on. Depending on the value of [`gc_protect_expires_after`](/docs/v23.2/create-changefeed#gc-protect-expire), this can lead to data accumulation. Once `gc_protect_expires_after` elapses, the protected timestamp record will be released and the changefeed job will be canceled. See [Garbage collection and changefeeds](/docs/v23.2/protect-changefeed-data) for more detail.

To avoid these issues, use the `jobs.{job_type}.currently_paused` metric to track the number of jobs (for each job type) that are currently considered paused.

You can monitor protected timestamps relating to particular CockroachDB jobs with the following metrics:

- `jobs.{job_type}.protected_age_sec` tracks the oldest protected timestamp record protecting `{job_type}` jobs. As this metric increases, garbage accumulation increases. Garbage collection will not progress on a table, database, or cluster if the protected timestamp record is present.
- `jobs.{job_type}.protected_record_count` tracks the number of protected timestamp records held by `{job_type}` jobs.

For a full list of the available job types, access your cluster's [`/_status/vars`](/docs/v23.2/monitoring-and-alerting#prometheus-endpoint) endpoint.

See the following pages for details on metrics:

- [Monitor and Debug Changefeeds](/docs/v23.2/monitor-and-debug-changefeeds)
- [Backup and Restore Monitoring](/docs/v23.2/backup-and-restore-monitoring)
- [Metrics](/docs/v23.2/metrics)

## Examples

### Pause a single job

```sql
> SHOW JOBS;
```

```text
      job_id     |  job_type |               description                      |...
-----------------+-----------+------------------------------------------------+...
  27536791415282 |  RESTORE  | RESTORE db.* FROM 'azure-blob://backup/db/tbl' |...
```

```sql
> PAUSE JOB 27536791415282;
```

### Pause multiple jobs

To pause multiple jobs, nest a [`SELECT` clause](/docs/v23.2/select-clause) that retrieves `job_id`(s) inside the `PAUSE JOBS` statement:

```sql
> PAUSE JOBS (WITH x AS (SHOW JOBS) SELECT job_id FROM x
      WHERE user_name = 'maxroach');
```

All jobs created by `maxroach` will be paused.

### Pause by job type

To pause all jobs by the type of job, use the `PAUSE ALL {job} JOBS` statement. You can pause all `BACKUP`, `RESTORE`, `CHANGEFEED`, `IMPORT` jobs using this statement, for example:

```sql
PAUSE ALL BACKUP JOBS;
```

### Pause automatic table statistics jobs

```sql
> SHOW AUTOMATIC JOBS;
```

```text
        job_id       |       job_type      |                    description                      |...
---------------------+---------------------+-----------------------------------------------------+...
  438235476849557505 |  AUTO CREATE STATS  | Table statistics refresh for defaultdb.public.users |...
(1 row)
```

```sql
> PAUSE JOB 438235476849557505;
```

To permanently disable automatic table statistics jobs, disable the `sql.stats.automatic_collection.enabled` [cluster setting](/docs/v23.2/cluster-settings):

```sql
> SET CLUSTER SETTING sql.stats.automatic_collection.enabled = false;
```

### Pause jobs for a schedule

To pause jobs for a specific [backup schedule](/docs/v23.2/create-schedule-for-backup), use the schedule's `id`:

```sql
> PAUSE JOBS FOR SCHEDULE 590204387299262465;
```

```text
PAUSE JOBS FOR SCHEDULES 1
```

You can also pause multiple schedules by nesting a [`SELECT` clause](/docs/v23.2/select-clause) that retrieves `id`(s) inside the `PAUSE JOBS` statement:

```sql
> PAUSE JOBS FOR SCHEDULES WITH x AS (SHOW SCHEDULES) SELECT id FROM x WHERE label = 'test_schedule';
```

```text
PAUSE JOBS FOR SCHEDULES 2
```
