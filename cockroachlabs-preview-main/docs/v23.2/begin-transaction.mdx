The `BEGIN` [statement](/docs/v23.2/sql-statements) initiates a [transaction](/docs/v23.2/transactions), which either successfully executes all of the statements it contains or none at all.

<Note>
When running under the default [`SERIALIZABLE`](/docs/v23.2/demo-serializable) isolation level, your application should [use a retry loop to handle transaction retry errors](/docs/v23.2/query-behavior-troubleshooting#transaction-retry-errors) that can occur under [contention](/docs/v23.2/performance-best-practices-overview#transaction-contention).
</Note>

## Required privileges

No [privileges](/docs/v23.2/security-reference/authorization#managing-privileges) are required to initiate a transaction. However, privileges are required for each statement within a transaction.

## Aliases

In CockroachDB, the following are aliases for the `BEGIN` statement:

- `BEGIN TRANSACTION`
- `START TRANSACTION`

## Examples

### Begin a transaction

#### Use default settings

Without modifying the `BEGIN` statement, the transaction uses `SERIALIZABLE` isolation and `NORMAL` priority.

```sql
> BEGIN;
```

```sql
> SAVEPOINT cockroach_restart;
```

```sql
> UPDATE products SET inventory = 0 WHERE sku = '8675309';
```

```sql
> INSERT INTO orders (customer, sku, status) VALUES (1001, '8675309', 'new');
```

```sql
> RELEASE SAVEPOINT cockroach_restart;
```

```sql
> COMMIT;
```

<Warning>
This example assumes you're using [client-side retry handling](/docs/v23.2/transaction-retry-error-reference#client-side-retry-handling).
</Warning>

#### Change isolation level

You can set the transaction isolation level to [`SERIALIZABLE`](/docs/v23.2/demo-serializable) or [`READ COMMITTED`](/docs/v23.2/read-committed#enable-read-committed-isolation).

If not specified, transactions use the value of the current session's [`default_transaction_isolation`](/docs/v23.2/session-variables#default-transaction-isolation) variable.

```sql
BEGIN TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

#### Change priority

You can set a transaction's priority to `LOW` or `HIGH`.

```sql
> BEGIN PRIORITY HIGH;
```

```sql
> SAVEPOINT cockroach_restart;
```

```sql
> UPDATE products SET inventory = 0 WHERE sku = '8675309';
```

```sql
> INSERT INTO orders (customer, sku, status) VALUES (1001, '8675309', 'new');
```

```sql
> RELEASE SAVEPOINT cockroach_restart;
```

```sql
> COMMIT;
```

You can also set a transaction's priority with [`SET TRANSACTION`](/docs/v23.2/set-transaction).

<Warning>
This example assumes you're using [client-side retry handling](/docs/v23.2/transaction-retry-error-reference#client-side-retry-handling).
</Warning>

### Use the `AS OF SYSTEM TIME` option

You can execute the transaction using the database contents "as of" a specified time in the past.

```sql
> BEGIN AS OF SYSTEM TIME '2019-04-09 18:02:52.0+00:00';
```

```sql
> SELECT * FROM orders;
```

```sql
> SELECT * FROM products;
```

```sql
> COMMIT;
```

<Tip>
You can also use the [`SET TRANSACTION`](/docs/v23.2/set-transaction#use-the-as-of-system-time-option) statement inside the transaction to achieve the same results. This syntax is easier to use from [drivers and ORMs](/docs/v23.2/install-client-drivers).
</Tip>

### Begin a transaction with automatic retries

CockroachDB will [automatically retry](/docs/v23.2/transactions#transaction-retries) all transactions that contain both `BEGIN` and `COMMIT` in the same batch. Batching is controlled by your driver or client's behavior, but means that CockroachDB receives all of the statements as a single unit, instead of a number of requests.

From the perspective of CockroachDB, a transaction sent as a batch looks like this:

```sql
> BEGIN;

> DELETE FROM customers WHERE id = 1;

> DELETE orders WHERE customer = 1;

> COMMIT;
```

However, in your application's code, batched transactions are often just multiple statements sent at once. For example, in Go, this transaction would sent as a single batch (and automatically retried):

```go
db.Exec(
  "BEGIN;

  DELETE FROM customers WHERE id = 1;

  DELETE orders WHERE customer = 1;

  COMMIT;"
)
```

Issuing statements this way signals to CockroachDB that you do not need to change any of the statement's values if the transaction doesn't immediately succeed, so it can continually retry the transaction until it's accepted.
