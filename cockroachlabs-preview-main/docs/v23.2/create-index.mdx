The `CREATE INDEX` [statement](/docs/v23.2/sql-statements) creates an index for a table. [Indexes](/docs/v23.2/indexes) improve your database's performance by helping SQL locate data without having to look through every row of a table.

Indexes are automatically created for a table's [`PRIMARY KEY`](/docs/v23.2/primary-key) and [`UNIQUE`](/docs/v23.2/unique) columns. When querying a table, CockroachDB uses the fastest index. For more information about that process, see [Index Selection in CockroachDB](https://www.cockroachlabs.com/blog/index-selection-cockroachdb-2).

The following types cannot be included in an index key, but can be stored (and used in a covered query) using the [`STORING` or `COVERING`](/docs/v23.2/create-index#store-columns) clause:

- [`JSONB`](/docs/v23.2/jsonb)
- [`ARRAY`](/docs/v23.2/array)
- The computed [`TUPLE`](/docs/v23.2/scalar-expressions#tuple-constructors) type, even if it is constructed from indexed fields

To create an index on the schemaless data in a [`JSONB`](/docs/v23.2/jsonb) column or on the data in an [`ARRAY`](/docs/v23.2/array), use a [GIN index](/docs/v23.2/inverted-indexes).

<Note>
The `CREATE INDEX` statement performs a schema change. For more information about how online schema changes work in CockroachDB, see [Online Schema Changes](/docs/v23.2/online-schema-changes).
</Note>

## Required privileges

The user must have the `CREATE` [privilege](/docs/v23.2/security-reference/authorization#managing-privileges) on the table.

## Examples

To follow along, run [`cockroach demo`](/docs/v23.2/cockroach-demo) to start a temporary, in-memory cluster with the [`movr`](/docs/v23.2/movr) sample dataset preloaded:

```shell
$ cockroach demo
```

### Create standard indexes

To create the most efficient indexes, we recommend reviewing:

- [Indexes: Best Practices](/docs/v23.2/indexes#best-practices)
- [Index Selection in CockroachDB](https://www.cockroachlabs.com/blog/index-selection-cockroachdb-2)

#### Single-column indexes

Single-column indexes sort the values of a single column.

```sql
> CREATE INDEX ON users (name);
```

Because each query can only use one index, single-column indexes are not typically as useful as multiple-column indexes.

#### Multiple-column indexes

Multiple-column indexes sort columns in the order you list them.

```sql
> CREATE INDEX ON users (name, city);
```

To create the most useful multiple-column indexes, we recommend reviewing our [best practices](/docs/v23.2/schema-design-indexes#best-practices).

#### Unique indexes

Unique indexes do not allow duplicate values among their columns.

```sql
> CREATE UNIQUE INDEX ON users (name, id);
```

This also applies the [`UNIQUE` constraint](/docs/v23.2/unique) at the table level, similar to [`ALTER TABLE`](/docs/v23.2/alter-table). The preceding example is equivalent to:

```sql
> ALTER TABLE users ADD CONSTRAINT users_name_id_key UNIQUE (name, id);
```

Primary key columns that are not specified within a unique index are automatically marked as [`STORING`](/docs/v23.2/indexes#storing-columns) in the [`information_schema.statistics`](/docs/v23.2/information-schema#statistics) table and in [`SHOW INDEX`](/docs/v23.2/show-index).

### Create GIN indexes

You can create [GIN indexes](/docs/v23.2/inverted-indexes) on schemaless data in a [`JSONB`](/docs/v23.2/jsonb) column.

```sql
> CREATE INDEX ON promo_codes USING GIN (rules);
```

The following syntax is equivalent:

```sql
> CREATE INVERTED INDEX ON promo_codes (rules);
```

### Create trigram indexes

You can create [trigram indexes](/docs/v23.2/trigram-indexes) on `STRING` columns by specifying the `gin_trgm_ops` or `gist_trgm_ops` opclass:

```sql
CREATE INDEX ON rides USING GIN (vehicle_city gin_trgm_ops);
```

The following syntax is equivalent:

```sql
CREATE INVERTED INDEX ON rides(vehicle_city gin_trgm_ops);
```

<Note>
GIN and GiST indexes are implemented identically on CockroachDB. `GIN` and `GIST` are therefore synonymous when defining a trigram index.
</Note>

### Create spatial indexes

You can create [spatial indexes](/docs/v23.2/spatial-indexes) on `GEOMETRY` and `GEOGRAPHY` columns. Spatial indexes are a special type of [GIN index](/docs/v23.2/inverted-indexes).

To create a spatial index on a `GEOMETRY` column:

```sql
CREATE INDEX geom_idx_1 ON some_spatial_table USING GIST(geom);
```

Unlike GIN indexes, spatial indexes do not support an alternate `CREATE INVERTED INDEX...` syntax. Only the syntax shown here is supported.

For advanced users, there are a number of [spatial index tuning parameters](/docs/v23.2/spatial-indexes#create-a-spatial-index-that-uses-all-of-the-tuning-parameters) that can be passed in using the syntax `WITH (var1=val1, var2=val2)` as follows:

```sql
CREATE INDEX geom_idx_2
  ON some_spatial_table USING GIST(geom)
  WITH (s2_max_cells = 20, s2_max_level = 12, s2_level_mod = 3);
```

<Warning>
Most users should not change the default spatial index settings. There is a risk that you will get worse performance by changing the default settings. For more information, see [Spatial indexes](/docs/v23.2/spatial-indexes).
</Warning>

### Store columns

Storing a column improves the performance of queries that retrieve (but do not filter) its values.

```sql
> CREATE INDEX ON users (city) STORING (name);
```

However, to use stored columns, queries must filter another column in the same index. For example, SQL can retrieve `name` values from the above index only when a query's `WHERE` clause filters `city`.

<Note>
An index that stores all the columns needed by a query is also known as a *covering index* for that query. When a query has a covering index, CockroachDB can use that index directly instead of doing an "index join" with the primary index, which is likely to be slower.
</Note>

### Change column sort order

To sort columns in descending order, you must explicitly set the option when creating the index. (Ascending order is the default.)

```sql
> CREATE INDEX ON users (city DESC, name);
```

How a column is ordered in the index will affect the ordering of the index keys, and may affect the efficiency of queries that include an `ORDER BY` clause.

### Query specific indexes

Normally, CockroachDB selects the index that it calculates will scan the fewest rows. However, you can override that selection and specify the name of the index you want to use. To find the name, use [`SHOW INDEX`](/docs/v23.2/show-index).

```sql
> SHOW INDEX FROM users;
```

```text
  table_name |   index_name        | non_unique | seq_in_index | column_name | direction | storing | implicit
+------------+---------------------+------------+--------------+-------------+-----------+---------+----------+
  users      | users_pkey          |   false    |            1 | city        | ASC       |  false  |  false
  users      | users_pkey          |   false    |            2 | id          | ASC       |  false  |  false
  users      | users_pkey          |   false    |            3 | name        | N/A       |  true   |  false
  users      | users_pkey          |   false    |            4 | address     | N/A       |  true   |  false
  users      | users_pkey          |   false    |            5 | credit_card | N/A       |  true   |  false
  users      | users_city_name_idx |    true    |            1 | city        | DESC      |  false  |  false
  users      | users_city_name_idx |    true    |            2 | name        | ASC       |  false  |  false
  users      | users_city_name_idx |    true    |            3 | id          | ASC       |  false  |   true
(8 rows)
```

```sql
> SELECT name FROM users@users_name_idx WHERE city='new york';
```

```text
        name
+------------------+
  Catherine Nelson
  Devin Jordan
  James Hamilton
  Judy White
  Robert Murphy
(5 rows)
```

You can use the `@primary` alias to use the table's primary key in your query if no secondary index explicitly named `primary` exists on that table.

### Create a hash-sharded secondary index

We [discourage indexing on sequential keys](/docs/v23.2/schema-design-indexes#best-practices). If a table **must** be indexed on sequential keys, use [hash-sharded indexes](/docs/v23.2/hash-sharded-indexes). Hash-sharded indexes distribute sequential traffic uniformly across ranges, eliminating single-range [hot spots](/docs/v23.2/performance-best-practices-overview#hot-spots) and improving write performance on sequentially-keyed indexes at a small cost to read performance.

Let's assume the `events` table already exists:

```sql
> CREATE TABLE events (
    product_id INT8,
    owner UUID,
    serial_number VARCHAR,
    event_id UUID,
    ts TIMESTAMP,
    data JSONB,
    PRIMARY KEY (product_id, owner, serial_number, ts, event_id)
);
```

You can create a hash-sharded index on an existing table:

```sql
> CREATE INDEX ON events(ts) USING HASH;
```

```sql
> SHOW INDEX FROM events;
```

```text
  table_name |  index_name   | non_unique | seq_in_index |        column_name        | direction | storing | implicit
-------------+---------------+------------+--------------+---------------------------+-----------+---------+-----------
  events     | events_pkey   |   false    |            1 | product_id                | ASC       |  false  |  false
  events     | events_pkey   |   false    |            2 | owner                     | ASC       |  false  |  false
  events     | events_pkey   |   false    |            3 | serial_number             | ASC       |  false  |  false
  events     | events_pkey   |   false    |            4 | ts                        | ASC       |  false  |  false
  events     | events_pkey   |   false    |            5 | event_id                  | ASC       |  false  |  false
  events     | events_pkey   |   false    |            6 | data                      | N/A       |  true   |  false
  events     | events_ts_idx |    true    |            1 | crdb_internal_ts_shard_16 | ASC       |  false  |   true
  events     | events_ts_idx |    true    |            2 | ts                        | ASC       |  false  |  false
  events     | events_ts_idx |    true    |            3 | product_id                | ASC       |  false  |   true
  events     | events_ts_idx |    true    |            4 | owner                     | ASC       |  false  |   true
  events     | events_ts_idx |    true    |            5 | serial_number             | ASC       |  false  |   true
  events     | events_ts_idx |    true    |            6 | event_id                  | ASC       |  false  |   true
(12 rows)
```
