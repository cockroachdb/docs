The `cockroach debug zip` [command](/docs/v26.1/cockroach-commands) connects to your cluster and gathers information from each active node into a single `.zip` file (inactive nodes are not included). For details on the `.zip` contents, see [Files](#files).

You can use the [`cockroach debug merge-logs`](/docs/v26.1/cockroach-debug-merge-logs) command in conjunction with `cockroach debug zip` to merge the collected logs into one file, making them easier to parse.

<Warning>
The files produced by `cockroach debug zip` can contain highly [sensitive, personally-identifiable information (PII)](/docs/v26.1/configure-logs#redact-logs), such as usernames, hashed passwords, and possibly table data. Use the [`--redact`](#redact) flag to configure CockroachDB to redact sensitive data when generating the `.zip` file (excluding range keys) if intending to share it with Cockroach Labs.
</Warning>

## Details

### Use cases

<Warning>
`cockroach debug zip` is an expensive operation and impacts cluster performance.

Only use this command as an emergency measure under the guidance of Cockroach Labs.

Particularly fetching stack traces for all goroutines is a "stop-the-world" operation, which can momentarily but significantly increase SQL service latency. Exclude these goroutine stacks by using the [`--include-goroutine-stacks=false` flag](#include-goroutine-stacks).
</Warning>

There are two scenarios in which `debug zip` is useful:

- If you experience severe or difficult-to-reproduce issues with your cluster, Cockroach Labs might ask you to send us your cluster's debugging information using `cockroach debug zip`. We recommend reducing the `*.zip` file size by only [retrieving debugging information for the relevant time range](#generate-a-debug-zip-file-for-a-time-range) of the issue by using the `--files-from`, and/or `--files-until` [flags](#flags).
- To collect all of your nodes' logs, which you can then parse to locate issues. You can optionally use the [flags](#flags) to [retrieve only the log files](#generate-a-debug-zip-file-with-logs-only). For more information about logs, see [Logging](/docs/v26.1/logging-overview). Also note:
    - Nodes that are currently [down](/docs/v26.1/cluster-setup-troubleshooting#node-liveness-issues) cannot deliver their logs over the network. For these nodes, you must log on to the machine where the `cockroach` process would otherwise be running, and gather the files manually.
    - Nodes that are currently up but disconnected from other nodes (e.g., because of a [network partition](/docs/v26.1/cluster-setup-troubleshooting#network-partition) ) may not be able to respond to `debug zip` requests forwarded by other nodes, but can still respond to requests for data when asked directly. In such situations, we recommend using the [`--host` flag](#client-connection) to point `debug zip` at each of the disconnected nodes until data has been gathered for the entire cluster.

### Files

`cockroach debug zip` collects log files, heap profiles, CPU profiles, and goroutine dumps from the last 48 hours, by default.

<Tip>
These files can greatly increase the size of the `cockroach debug zip` output. To limit the `.zip` file size for a large cluster, we recommend first experimenting with [`cockroach debug list-files`](/docs/v26.1/cockroach-debug-list-files) and then using [flags](#flags) to filter the files.
</Tip>

The following files collected by `cockroach debug zip`, which are found in the individual node directories, can be filtered using the `--exclude-files`, `--include-files`, `--files-from`, and/or `--files-until` [flags](#flags):

<table><thead><tr><th>Information</th><th>Filename</th></tr></thead><tbody><tr><td><a href="/docs/v26.1/configure-logs#log-file-naming">Log files</a></td><td><code>cockroach-&#123;log-file-group&#125;.&#123;host&#125;.&#123;user&#125;.&#123;start timestamp in UTC&#125;.&#123;process ID&#125;.log</code></td></tr><tr><td>Goroutine dumps</td><td><code>goroutine&#95;dump.&#123;date-and-time&#125;.&#123;metadata&#125;.double&#95;since&#95;last&#95;dump.&#123;metadata&#125;.txt.gz</code></td></tr><tr><td>Heap profiles</td><td><code>memprof.&#123;date-and-time&#125;.&#123;heapsize&#125;.pprof</code></td></tr><tr><td>Memory statistics</td><td><code>memstats.&#123;date-and-time&#125;.&#123;heapsize&#125;.txt</code></td></tr><tr><td>CPU profiles</td><td><code>cpuprof.&#123;date-and-time&#125;</code></td></tr><tr><td><a href="/docs/v26.1/cluster-setup-troubleshooting#out-of-memory-oom-crash">Active query dumps</a></td><td><code>activequeryprof.&#123;date-and-time&#125;.csv</code></td></tr></tbody></table>

The following information is also contained in the `.zip` file, and cannot be filtered:

- System tables. The following system tables are not included:
    - `system.users`
    - `system.web_sessions`
    - `system.join_tokens`
    - `system.comments`
    - `system.ui`
    - `system.zones`
    - `system.statement_bundle_chunks`
    - `system.statement_statistics`
    - `system.transaction_statistics`
- Cluster events
- Database details
- Schema change events
- Database, table, node, and range lists
- Node details
- Node liveness
- Gossip data
- Stack traces
- Range details
- Jobs
- [Cluster Settings](/docs/v26.1/cluster-settings)
- [Metrics](/docs/v26.1/metrics)
- CPU profiles
- A script ( `hot-ranges.sh` ) that summarizes the hottest ranges (ranges receiving a high number of reads or writes)

## Subcommands

While the `cockroach debug` command has a few subcommands, users are expected to use only the [`zip`](/docs/v26.1/cockroach-debug-zip), [`encryption-active-key`](/docs/v26.1/cockroach-debug-encryption-active-key), [`merge-logs`](/docs/v26.1/cockroach-debug-merge-logs), [`list-files`](/docs/v26.1/cockroach-debug-list-files), [`tsdump`](/docs/v26.1/cockroach-debug-tsdump), and [`ballast`](/docs/v26.1/cockroach-debug-ballast) subcommands.

We recommend using the [`encryption-decrypt`](/docs/v26.1/cockroach-debug-encryption-decrypt) and [`job-trace`](/docs/v26.1/cockroach-debug-job-trace) subcommands only when directed by the [Cockroach Labs support team](/docs/v26.1/support-resources).

The other `debug` subcommands are useful only to Cockroach Labs. Output of `debug` commands may contain sensitive or secret information.

## Synopsis

```shell
$ cockroach debug zip {ZIP file destination} {flags}
```

<Note>
The following [flags](#flags) must apply to an active CockroachDB node. If no nodes are live, you must [start at least one node](/docs/v26.1/cockroach-start).
</Note>

### Logging

By default, this command logs messages to `stderr`. This includes events with `WARNING` [severity](/docs/v26.1/logging#logging-levels-severities) and higher.

If you need to troubleshoot this command's behavior, you can [customize its logging behavior](/docs/v26.1/configure-logs).

## Examples

### Generate a debug zip file

Generate the debug zip file for an insecure cluster:

```shell
$ cockroach debug zip ./cockroach-data/logs/debug.zip --insecure --host=200.100.50.25
```

Generate the debug zip file for a secure cluster:

```shell
$ cockroach debug zip ./cockroach-data/logs/debug.zip --host=200.100.50.25
```

<Note>
Secure examples assume you have the appropriate certificates in the default certificate directory, `${HOME}/.cockroach-certs/`.
</Note>

### Generate a debug zip file for a time range

Generate a debug zip file containing only debugging information for a specified time range:

```shell
$ cockroach debug zip ./cockroach-data/logs/debug.zip --files-from='2023-10-03 13:30' --files-until='2023-10-03 14:30'
```

### Generate a debug zip file with logs only

Generate a debug zip file containing only log files:

```shell
$ cockroach debug zip ./cockroach-data/logs/debug.zip --include-files=*.log
```

### Redact sensitive information

#### Log redaction

Example of a log string without [`--redact`](#redact) enabled:

```text
server/server.go:1423 ⋮ password of user ‹admin› was set to ‹"s3cr34?!@x_"›
```

Enable log redaction:

```shell
$ cockroach debug zip ./cockroach-data/logs/debug.zip --redact --insecure --host=200.100.50.25
```

```text
server/server.go:1423 ⋮ password of user ‹×› was set to ‹×›
```

#### Cluster settings redaction

Example cluster settings in `crdb_internal.cluster_settings.txt` without [`--redact`](#redact) enabled:

```text
variable                          value                  type public sensitive reportable description                                   default_value origin
...
cluster.organization              Cockroach Labs Testing s    t      f         f          organization name                                           override
...
server.identity_map.configuration <redacted>             s    t      t         f          system-identity to database-username mappings               default
```

`server.identity_map.configuration` is always redacted, since `sensitive` equals `true`.

Enable log redaction:

```shell
$ cockroach debug zip ./cockroach-data/logs/debug.zip --redact --insecure --host=200.100.50.25
```

Cluster settings in `crdb_internal.cluster_settings.txt` with [`--redact`](#redact) enabled:

```text
variable                          value      type public sensitive reportable description                                    default_value origin
...
cluster.organization              <redacted> s    t      f         f          organization name                                            override
...
server.identity_map.configuration <redacted> s    t      t         f          system-identity to database-username mappings                default
```

`server.identity_map.configuration` is still redacted. `cluster.organization` is now redacted since `reportable` equals `false` and the `Cockroach Labs Testing` value is not the default value (in this case, the empty string).

#### Hostname and IP address redaction

Example of `status.json` without hostname and IP address redaction enabled:

```text
{
  "node_id": 1,
  "address": {
    "network_field": "tcp",
    "address_field": "200.100.50.25:26257"
  },
  "sql_address": {
    "network_field": "tcp",
    "address_field": "200.100.50.25:26257"
  }
}
```

Enable hostname and IP address redaction with the [`debug.zip.redact_addresses.enabled`](/docs/v26.1/cluster-settings#setting-debug-zip-redact-addresses-enabled) cluster setting:

```sql
SET CLUSTER SETTING debug.zip.redact_addresses.enabled = true;
```

<Note>
Some hostnames and IP addresses in the `nodes.json` and `gossip.json` files are **never** redacted, even when `debug.zip.redact_addresses.enabled` is enabled.
</Note>

Generate `.zip` with [`--redact`](#redact) enabled:

```shell
cockroach debug zip ./cockroach-data/logs/debug.zip --redact --insecure --host=200.100.50.25
```

`status.json` with hostname and IP address redaction:

```text
{
  "node_id": 1,
  "address": {
    "network_field": "tcp",
    "address_field": "‹×›"
  },
  "sql_address": {
    "network_field": "tcp",
    "address_field": "‹×›"
  }
}
```
