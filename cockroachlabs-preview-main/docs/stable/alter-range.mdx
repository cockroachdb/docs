The `ALTER RANGE` [statement](/docs/v26.1/sql-statements) applies a [schema change](/docs/v26.1/online-schema-changes) to a range.

## Required privileges

Refer to the respective [subcommands](#subcommands).

## Subcommands

### `CONFIGURE ZONE`

`ALTER RANGE... CONFIGURE ZONE` is used to add, modify, reset, or remove [replication zones](/docs/v26.1/configure-replication-zones) for a range. To view details about existing replication zones, see [`SHOW ZONE CONFIGURATIONS`](/docs/v26.1/show-zone-configurations).

You can use replication zones to control the number and location of replicas for specific sets of data, both when replicas are first added and when they are rebalanced to maintain cluster equilibrium.

For instructions showing how to troubleshoot replication zones that may be misconfigured, see [Troubleshoot Replication Zone Configurations](/docs/v26.1/troubleshoot-replication-zones).

#### Required privileges

The user must be a member of the [`admin` role](/docs/v26.1/security-reference/authorization#admin-role) or have been granted [`CREATE`](/docs/v26.1/security-reference/authorization#supported-privileges) or [`ZONECONFIG`](/docs/v26.1/security-reference/authorization#supported-privileges) privileges. To configure [`system` objects](/docs/v26.1/configure-replication-zones#for-system-data), the user must be a member of the `admin` role.

#### Parameters

<table><thead><tr><th>Parameter</th><th>Description</th></tr></thead><tbody><tr><td><code>variable</code></td><td>The name of the <a href="/docs/v26.1/configure-replication-zones#replication-zone-variables">replication zone variable</a> to change.</td></tr><tr><td><code>value</code></td><td>The value of the <a href="/docs/v26.1/configure-replication-zones#replication-zone-variables">replication zone variable</a> to change.</td></tr><tr><td><code>DISCARD</code></td><td>Remove a replication zone.</td></tr></tbody></table>

For usage, see [Synopsis](#synopsis).

### `RELOCATE`

`ALTER RANGE... RELOCATE` is used to move a lease or [replica](/docs/v26.1/architecture/overview#architecture-replica) between [stores](/docs/v26.1/cockroach-start#store). This is helpful in an emergency situation to relocate data in the cluster.

<Warning>
Most users should not need to use this statement; it is for use in emergency situations. If you are in an emergency situation where you think using this statement may help, Cockroach Labs recommends contacting [support](/docs/v26.1/support-resources).
</Warning>

#### Required privileges

To alter a range and move a lease or replica between stores, the user must have one of the following:

- Membership to the [`admin`](/docs/v26.1/security-reference/authorization#admin-role) role for the cluster.

#### Parameters

<table><thead><tr><th>Parameter</th><th>Description</th></tr></thead><tbody><tr><td><code>LEASE</code></td><td>Move <a href="/docs/v26.1/architecture/replication-layer#leases">leases</a>.</td></tr><tr><td><code>VOTERS</code></td><td>Move <a href="/docs/v26.1/architecture/overview#architecture-replica">voting replicas</a> only.</td></tr><tr><td><code>NONVOTERS</code></td><td>Move <a href="/docs/v26.1/architecture/replication-layer#non-voting-replicas">non-voting replicas</a> only.</td></tr><tr><td><code>FROM a&#95;expr</code></td><td>Move a replica from a store ID.</td></tr><tr><td><code>TO a&#95;expr</code></td><td>Move a lease or replica to a new store ID.</td></tr><tr><td><code>select&#95;stmt</code></td><td>A <a href="/docs/v26.1/selection-queries">selection query</a> that produces one or more rows.</td></tr></tbody></table>

## Examples

### Configure replication zones

#### Create a replication zone for a system range

In addition to the databases and tables that are visible via the SQL interface, CockroachDB stores internal data in what are called system ranges. CockroachDB comes with pre-configured replication zones for some of these ranges:

<table><thead><tr><th>Target Name</th><th>Description</th></tr></thead><tbody><tr><td><code>meta</code></td><td>The "meta" ranges contain the authoritative information about the location of all data in the cluster.<br/><br/>These ranges must retain a majority of replicas for the cluster as a whole to remain available and historical queries are never run on them, so CockroachDB comes with a <strong>pre-configured</strong><code>meta</code> replication zone with <code>num&#95;replicas</code> set to 5 to make these ranges more resilient to node failure and a lower-than-default <code>gc.ttlseconds</code> to keep these ranges smaller for reliable performance.<br/><br/>If your cluster is running in multiple datacenters, it's a best practice to configure the meta ranges to have a copy in each datacenter.</td></tr><tr><td><code>liveness</code></td><td>The "liveness" range contains the authoritative information about which nodes are live at any given time.<br/><br/>These ranges must retain a majority of replicas for the cluster as a whole to remain available and historical queries are never run on them, so CockroachDB comes with a <strong>pre-configured</strong><code>liveness</code> replication zone with <code>num&#95;replicas</code> set to 5 to make these ranges more resilient to node failure and a lower-than-default <code>gc.ttlseconds</code> to keep these ranges smaller for reliable performance.</td></tr><tr><td><code>system</code></td><td>There are system ranges for a variety of other important internal data, including information needed to allocate new table IDs and track the status of a cluster's nodes.<br/><br/>These ranges must retain a majority of replicas for the cluster as a whole to remain available, so CockroachDB comes with a <strong>pre-configured</strong><code>system</code> replication zone with <code>num&#95;replicas</code> set to 5 to make these ranges more resilient to node failure.</td></tr><tr><td><code>timeseries</code></td><td>The "timeseries" ranges contain monitoring data about the cluster that powers the graphs in CockroachDB's DB Console. If necessary, you can add a <code>timeseries</code> replication zone to control the replication of this data.</td></tr></tbody></table>

<Warning>
Use caution when editing replication zones for system ranges, as they could cause some (or all) parts of your cluster to stop working.
</Warning>

To control replication for one of the above sets of system ranges, use the [`ALTER RANGE... CONFIGURE ZONE`](/docs/v26.1/alter-range#configure-zone) statement to define the relevant values (other values will be inherited from the parent zone):

```sql
> ALTER RANGE meta CONFIGURE ZONE USING num_replicas = 7;
```

```text
CONFIGURE ZONE 1
```

```sql
> SHOW ZONE CONFIGURATION FROM RANGE meta;
```

```text
    target   |            raw_config_sql
+------------+---------------------------------------+
  RANGE meta | ALTER RANGE meta CONFIGURE ZONE USING
             |     range_min_bytes = 134217728,
             |     range_max_bytes = 536870912,
             |     gc.ttlseconds = 3600,
             |     num_replicas = 7,
             |     constraints = '[]',
             |     lease_preferences = '[]'
(1 row)
```

#### Find range ID and leaseholder information

To use `ALTER RANGE... RELOCATE`, you need to know how to find the range ID, leaseholder, and other information for a [table](/docs/v26.1/show-ranges#show-ranges-for-a-table), [index](/docs/v26.1/show-ranges#show-ranges-for-an-index), or [database](/docs/v26.1/show-ranges#show-ranges-for-a-database). You can find this information using the [`SHOW RANGES`](/docs/v26.1/show-ranges) statement.

For example, to get all range IDs, leaseholder store IDs, and leaseholder localities for the [`movr.users`](/docs/v26.1/movr) table, use the following query:

```sql
WITH user_info AS (SHOW RANGES FROM TABLE users WITH DETAILS) SELECT range_id, lease_holder, lease_holder_locality FROM user_info;
```

#### Move all of a range's non-voting replicas from one store to another store

To move a range's [non-voting replicas](/docs/v26.1/architecture/replication-layer#non-voting-replicas), use the statement below.

<Note>
This statement will only have an effect on clusters that have non-voting replicas configured, such as [multiregion clusters](/docs/v26.1/multiregion-overview). If your cluster is not a multiregion cluster, it doesn't do anything, and will display errors in the `result` field as shown below.
</Note>

```sql
ALTER RANGE RELOCATE NONVOTERS FROM 7 TO 2 FOR SELECT range_id from [SHOW RANGES FROM TABLE users WITH DETAILS];
```
