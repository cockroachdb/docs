The `DROP TABLE` [statement](/docs/v25.2/sql-statements) removes a table and all its indexes from a database.

<Note>
The `DROP TABLE` statement performs a schema change. For more information about how online schema changes work in CockroachDB, see [Online Schema Changes](/docs/v25.2/online-schema-changes).
</Note>

## Required privileges

The user must have the `DROP` [privilege](/docs/v25.2/security-reference/authorization#managing-privileges) on the specified table(s). If `CASCADE` is used, the user must have the privileges required to drop each dependent object as well.

## Viewing schema changes

This schema change statement is registered as a job. You can view long-running jobs with [`SHOW JOBS`](/docs/v25.2/show-jobs).

## Examples

#### Setup

To follow along, run [`cockroach demo`](/docs/v25.2/cockroach-demo) to start a temporary, in-memory cluster with the [`movr`](/docs/v25.2/movr) sample dataset preloaded:

```shell
$ cockroach demo
```

### Remove a table (no dependencies)

In this example, other objects do not depend on the table being dropped.

```sql
> SHOW TABLES FROM movr;
```

```text
  schema_name |         table_name         | type  | estimated_row_count
--------------+----------------------------+-------+----------------------
  public      | promo_codes                | table |                1000
  public      | rides                      | table |                 500
  public      | user_promo_codes           | table |                   0
  public      | users                      | table |                  50
  public      | vehicle_location_histories | table |                1000
  public      | vehicles                   | table |                  15
(6 rows)
```

```sql
> DROP TABLE promo_codes;
```

```text
DROP TABLE
```

```sql
> SHOW TABLES FROM movr;
```

```text
  schema_name |         table_name         | type  | estimated_row_count
```

### Remove a table and dependent objects with `CASCADE`

In this example, a [foreign key](/docs/v25.2/foreign-key) from a different table references the table being dropped. Therefore, it's only possible to drop the table while simultaneously dropping the dependent foreign key constraint using `CASCADE`.

<Warning>
`CASCADE` drops *all* dependent objects without listing them, which can lead to inadvertent and difficult-to-recover losses. To avoid potential harm, we recommend dropping objects individually in most cases.
</Warning>

```sql
> SHOW TABLES FROM movr;
```

```text
  schema_name |         table_name         | type  | estimated_row_count
--------------+----------------------------+-------+----------------------
  public      | rides                      | table |                 500
  public      | user_promo_codes           | table |                   0
  public      | users                      | table |                  50
  public      | vehicle_location_histories | table |                1000
  public      | vehicles                   | table |                  15
(5 rows)
```

```sql
> DROP TABLE users;
```

```text
pq: "users" is referenced by foreign key from table "vehicles"
```

To see how `users` is referenced from `vehicles`, you can use the [`SHOW CREATE`](/docs/v25.2/show-create) statement. `SHOW CREATE` shows how the columns in a table are created, including data types, default values, indexes, and constraints.

```sql
> SHOW CREATE TABLE vehicles;
```

```text
  table_name |                                         create_statement
-------------+---------------------------------------------------------------------------------------------------
  vehicles   | CREATE TABLE public.vehicles (
             |     id UUID NOT NULL,
             |     city VARCHAR NOT NULL,
             |     type VARCHAR NULL,
             |     owner_id UUID NULL,
             |     creation_time TIMESTAMP NULL,
             |     status VARCHAR NULL,
             |     current_location VARCHAR NULL,
             |     ext JSONB NULL,
             |     CONSTRAINT vehicles_pkey PRIMARY KEY (city ASC, id ASC),
             |     CONSTRAINT fk_city_ref_users FOREIGN KEY (city, owner_id) REFERENCES public.users(city, id),
             |     INDEX vehicles_auto_index_fk_city_ref_users (city ASC, owner_id ASC),
             |     FAMILY "primary" (id, city, type, owner_id, creation_time, status, current_location, ext)
             | )
(1 row)
```

```sql
> DROP TABLE users CASCADE;
```

```text
DROP TABLE
```

```sql
> SHOW TABLES FROM movr;
```

```text
  schema_name |         table_name         | type  | estimated_row_count
--------------+----------------------------+-------+----------------------

  public      | user_promo_codes           | table |                   0
  public      | vehicle_location_histories | table |                1000
  public      | vehicles                   | table |                  15
(4 rows)
```
