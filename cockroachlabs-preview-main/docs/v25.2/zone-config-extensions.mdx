Multi-Region *Zone Config Extensions* are a customization tool for advanced users to persistently modify the configuration generated by the standard [multi-region SQL abstractions](/docs/v25.2/multiregion-overview) on a per-region basis.

They are controlled using the [`ALTER DATABASE... ALTER LOCALITY`](/docs/v25.2/alter-database#alter-locality) statement.

Use cases for Zone Config Extensions include:

- Failover behavior
- Regions with stale read access
- Quorum placement
- Data domiciling

We strongly recommend that most users use the [default multi-region SQL abstractions](/docs/v25.2/multiregion-overview) as much as possible. However, if you need additional customizations, using Zone Config Extensions is better than writing your own low-level [zone configs](/docs/v25.2/configure-replication-zones) for the following reasons:

- Zone Config Extensions are persistent and kept up-to-date. Unlike raw zone config modification, which must be fully overwritten on each multi-region configuration change in order to take effect, Zone Config Extensions stay in place alongside the high-level [multi-region SQL configuration](/docs/v25.2/multiregion-overview). This is true even if you [add new tables](/docs/v25.2/create-table) or [regions](/docs/v25.2/alter-database#add-region), or if a table's [locality is changed](/docs/v25.2/alter-table#set-locality).
- They are validated against the configs already in place in a multi-region cluster. Unlike hand-written zone configs, they are designed to work properly with the configuration generated by the higher-level multi-region SQL statements. Specifically, this means:
    - [`REGIONAL`](/docs/v25.2/alter-database#parameters) extensions don't set [lease preferences](/docs/v25.2/configure-replication-zones#lease_preferences).
    - [`REGIONAL IN`](/docs/v25.2/alter-database#parameters) extensions don't set [lease preferences](/docs/v25.2/configure-replication-zones#lease_preferences), nor do they set [replica constraints](/docs/v25.2/configure-replication-zones#constraints) or [voter constraints](/docs/v25.2/configure-replication-zones#voter_constraints) that would conflict with the original [home region](/docs/v25.2/table-localities#regional-by-row-tables).
    - In no case will Zone Config Extensions change the [survival goals](/docs/v25.2/multiregion-overview#survival-goals) for the database.
- They are composable. You can use Zone Config Extensions to build up approximations of many higher-level features. For an example, see [Failover regions](/docs/v25.2/alter-database#failover-regions).
- They are region scoped. When a Zone Config Extension is attached to a [database region](/docs/v25.2/multiregion-overview#database-regions), any [`REGIONAL BY TABLE`](/docs/v25.2/table-localities#regional-tables) or [`REGIONAL BY ROW`](/docs/v25.2/table-localities#regional-by-row-tables) tables associated with that region will have their zone configurations updated according to the settings passed via the extension. As mentioned above, this updated config will also be persisted in the face of other multi-region configuration changes.
- They are locality scoped. You can specify a Zone Config Extension that only applies to tables with certain [localities](/docs/v25.2/multiregion-overview#table-localities). For example, to apply a Zone Config Extension to tables with the [`GLOBAL` locality](/docs/v25.2/table-localities#global-tables), use [`ALTER DATABASE... ALTER LOCALITY GLOBAL...`](/docs/v25.2/alter-database#alter-locality).

<Note>
Zone Config Extensions are a property of a [database region](/docs/v25.2/multiregion-overview#database-regions), *not* any particular [schema object](/docs/v25.2/schema-design-overview). This differs from raw [zone configurations](/docs/v25.2/configure-replication-zones), which may need to be assigned (and reassigned) to 10s or 100s of different schema objects on every configuration change.
</Note>

For instructions showing how to use Zone Config Extensions, see [Use Zone Config Extensions](/docs/v25.2/alter-database#use-zone-config-extensions).
