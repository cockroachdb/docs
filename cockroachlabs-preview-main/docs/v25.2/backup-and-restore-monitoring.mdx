CockroachDB includes metrics to monitor [backup](/docs/v25.2/backup), [restore](/docs/v25.2/restore), and [scheduled backup](/docs/v25.2/create-schedule-for-backup) jobs. You can use monitoring integrations to alert when there are anomalies, such as backups that have failed or restore jobs encountering a retryable error.

You can access the [Prometheus Endpoint](#prometheus-endpoint) to track and alert on backup and restore metrics. We recommend setting up monitoring to [alert](/docs/v25.2/monitoring-and-alerting#alerting-tools) when anomalies occur. You can then use the following SQL statements to inspect details relating to schedules, jobs, and backups:

- [`SHOW SCHEDULES`](/docs/v25.2/show-schedules)
- [`SHOW JOBS`](/docs/v25.2/show-jobs)
- [`SHOW BACKUP`](/docs/v25.2/show-backup)

<Note>
Metrics are reported per node. Therefore, it is necessary to retrieve metrics from every node in the cluster. For example, if you are monitoring whether a backup fails, it is necessary to track `scheduled_backup_failed` on each node.
</Note>

## Prometheus endpoint

You can access the [Prometheus endpoint](/docs/v25.2/monitoring-and-alerting#prometheus-endpoint) (`http://<host>:<http-port>/_status/vars`) for backup and restore metrics.

Refer to the [Monitor CockroachDB with Prometheus](/docs/v25.2/monitor-cockroachdb-with-prometheus) tutorial for guidance on installing and setting up Prometheus and Alertmanager to track metrics.

### Available metrics

We recommend the following guidelines:

- Use the `schedules.BACKUP.last-completed-time` metric to monitor the specific backup job or jobs you would use to recover from a disaster.
- Configure alerting on the `schedules.BACKUP.last-completed-time` metric to watch for cases where the timestamp has not moved forward as expected.
