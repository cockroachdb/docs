## Alter database schema objects

To change an existing object in a database schema, use an `ALTER` statement.

`ALTER` statements generally take the following form:

```text
ALTER {OBJECT_TYPE} {object_name} {SUBCOMMAND};
```

<table><thead><tr><th>Parameter</th><th>Description</th></tr></thead><tbody><tr><td><code>&#123;OBJECT&#95;TYPE&#125;</code></td><td>The type of the object.</td></tr><tr><td><code>&#123;object&#95;name&#125;</code></td><td>The name of the object.</td></tr><tr><td><code>&#123;SUBCOMMAND&#125;</code></td><td>The subcommand for the change that you would like to make.</td></tr></tbody></table>

For examples, see [below](#altering-objects-examples).

CockroachDB supports the following `ALTER` statements:

- [`ALTER DATABASE`](/docs/v25.2/alter-database)
- [`ALTER SCHEMA`](/docs/v25.2/alter-schema)
- [`ALTER TABLE`](/docs/v25.2/alter-table)
- [`ALTER INDEX`](/docs/v25.2/alter-index)
- [`ALTER VIEW`](/docs/v25.2/alter-view)
- [`ALTER SEQUENCE`](/docs/v25.2/alter-sequence)
- [`ALTER TYPE`](/docs/v25.2/alter-type)
- [`ALTER USER/ROLE`](/docs/v25.2/alter-user)

### Best practices for altering objects

- After you initialize a database schema, make any additional database schema changes in a separate set of changes (e.g., for the Cockroach SQL client, a separate `.sql` file; for Liquibase, a separate *changeset*).
- For `ALTER TABLE` statements, combine multiple subcommands in a single `ALTER TABLE` statement, where possible.
- Use a [database schema migration tool](/docs/v25.2/third-party-database-tools#schema-migration-tools) or the [CockroachDB SQL client](/docs/v25.2/cockroach-sql) instead of a [client library](/docs/v25.2/third-party-database-tools#drivers) to execute [database schema changes](/docs/v25.2/online-schema-changes).
- Review the [limitations of online schema changes](/docs/v25.2/online-schema-changes#known-limitations). CockroachDB [doesn't guarantee the atomicity of schema changes within transactions with multiple statements](/docs/v25.2/online-schema-changes#schema-changes-within-transactions).

    Cockroach Labs recommends that you perform schema changes outside explicit transactions. When a database [schema management tool](/docs/v25.2/third-party-database-tools#schema-migration-tools) manages transactions on your behalf, include one schema change operation per transaction.

### Altering objects examples

Suppose you want to make some changes to the `users` table that you created in [Create a Table](/docs/v25.2/schema-design-table). In specific, you want to do the following:

- Add a new `username` column.
- Change the columns in the table's primary key to `username` column and `email`.
- Move the table to the `abbey_schema` user-defined schema.
- Change the owner of the table to `abbey`.

The `ALTER TABLE` statement has subcommands for all of these changes:

- To add a new column, use the [`ADD COLUMN` subcommand](/docs/v25.2/alter-table#add-column).
- To change the primary key columns of a table, use the [`ALTER PRIMARY KEY` subcommand](/docs/v25.2/alter-table#alter-primary-key).
- To move a table to a different schema, use the [`SET SCHEMA`](/docs/v25.2/alter-table#set-schema) subcommand.
- To change the owner of a table, use the [`OWNER TO`](/docs/v25.2/alter-table#owner-to) subcommand.

Create a new `.sql` file for the changes that you plan to make to the table:

```shell
$ touch update_users_table.sql
```

Open `update_users_table.sql` in a text editor, and add the `ALTER TABLE` statement for adding the `username` column:

```sql
ALTER TABLE IF EXISTS movr.max_schema.users ADD COLUMN username STRING;
```

Under that first `ALTER TABLE` statement, add another `ALTER TABLE` statement for changing the primary key columns to `username` and `email`:

```sql
ALTER TABLE IF EXISTS movr.max_schema.users ALTER PRIMARY KEY USING COLUMNS (username, email);
```

In order to add a column to an existing table's primary key index, the column must have an existing [`NOT NULL` constraint](/docs/v25.2/not-null). Neither the `username` nor the `email` columns have `NOT NULL` constraints.

Add a `NOT NULL` constraint to the `ADD COLUMN` subcommand for `username`. In the same `ALTER TABLE` statement, add an [`ALTER COLUMN` subcommand](/docs/v25.2/alter-table#alter-column) to set the `NOT NULL` constraint on the `email` column:

```sql
ALTER TABLE IF EXISTS movr.max_schema.users
  ADD COLUMN username STRING NOT NULL,
  ALTER COLUMN email SET NOT NULL;
```

The file should now look something like this:

```sql
ALTER TABLE IF EXISTS movr.max_schema.users
  ADD COLUMN username STRING NOT NULL,
  ALTER COLUMN email SET NOT NULL;

ALTER TABLE IF EXISTS movr.max_schema.users ALTER PRIMARY KEY USING COLUMNS (username, email);
```

The remaining changes that you want to make will require `ALTER TABLE` statements with the `SET SCHEMA` and `OWNER TO` subcommands. An `ALTER TABLE... SET SCHEMA` statement will change the contents of two schemas, and an `ALTER TABLE... OWNER TO` statement will change the privileges of two users. To follow [authorization best practices](/docs/v25.2/security-reference/authorization#authorization-best-practices), you should execute any statements that change databases, user-defined schemas, or user privileges as a member of the `admin` role (e.g., as `root`).

Create a new `.sql` file for the remaining `ALTER TABLE` statements, to be executed by `root`:

```shell
$ touch update_users_owner.sql
```

Add the following statements to the file:

```sql
ALTER TABLE IF EXISTS movr.max_schema.users SET SCHEMA abbey_schema;

ALTER TABLE IF EXISTS movr.abbey_schema.users OWNER TO abbey;
```

To execute the statements in the `update_users_table.sql` file as `max`, run the following command:

```shell
$ cockroach sql \
--certs-dir={certs-directory} \
--user=max \
--database=movr \
-f update_users_table.sql
```

To execute the statements in the `update_users_owner.sql` file as `root`, run the following command:

```shell
$ cockroach sql \
--certs-dir={certs-directory} \
--user=root \
--database=movr \
-f update_users_owner.sql
```

The `users` table should now have a new column, a different primary key, a different schema, and a different owner.

You can verify with some `SHOW` statements:

```shell

```

## Drop database schema objects

To drop an object from a database schema, use a `DROP` statement.

`DROP` statements generally take the following form:

```text
DROP {OBJECT_TYPE} {object_name} CASCADE;
```

<table><thead><tr><th>Parameter</th><th>Description</th></tr></thead><tbody><tr><td><code>&#123;OBJECT&#95;TYPE&#125;</code></td><td>The type of the object.</td></tr><tr><td><code>&#123;object&#95;name&#125;</code></td><td>The name of the object.</td></tr><tr><td><code>&#123;CASCADE&#125;</code></td><td>An optional keyword that will drop all objects dependent on the object being dropped.</td></tr></tbody></table>

For examples, see [below](#altering-objects-examples).

CockroachDB supports the following `DROP` statements:

- [`DROP DATABASE`](/docs/v25.2/drop-database)
- [`DROP SCHEMA`](/docs/v25.2/drop-schema)
- [`DROP TABLE`](/docs/v25.2/drop-table)
- [`DROP INDEX`](/docs/v25.2/drop-index)
- [`DROP SEQUENCE`](/docs/v25.2/drop-sequence)
- [`DROP VIEW`](/docs/v25.2/drop-view)
- [`DROP USER/ROLE`](/docs/v25.2/drop-user)

<Note>
To drop columns and column constraints from a table, use the `DROP COLUMN` and `DROP CONSTRAINT` subcommands of the [`ALTER TABLE`](/docs/v25.2/alter-table) statement.
</Note>

### Drop best practices

- Check the contents and dependencies of the object that you want to drop before using the `CASCADE` option. `CASCADE` drops all of the contents of an object, and should be used sparingly after a schema has been initialized.

### Drop example

Suppose that you want to drop an index that isn't being used very much. In particular, you want to drop the index on `first_name` and `last_name` from the `users` table.

```shell
$ cockroach sql \
--certs-dir={certs-directory} \
--user=abbey \
--database=movr \
--execute="SHOW INDEXES FROM movr.abbey_schema.users; SHOW CREATE TABLE movr.abbey_schema.users;"
```

```text
  table_name |           index_name           | non_unique | seq_in_index | column_name | direction | storing | implicit  | visible
-------------+--------------------------------+------------+--------------+-------------+-----------+---------+-----------+---------
  users      | users_pkey                     |    f       |            1 | username    | ASC       |   f     |    f      |   t
  users      | users_pkey                     |    f       |            2 | email       | ASC       |   f     |    f      |   t
  users      | users_first_name_last_name_key |    f       |            1 | first_name  | ASC       |   f     |    f      |   t
  users      | users_first_name_last_name_key |    f       |            2 | last_name   | ASC       |   f     |    f      |   t
  users      | users_first_name_last_name_key |    f       |            3 | username    | ASC       |   f     |    t      |   t
  users      | users_first_name_last_name_key |    f       |            4 | email       | ASC       |   f     |    t      |   t
  users      | users_email_key                |    f       |            1 | email       | ASC       |   f     |    f      |   t
  users      | users_email_key                |    f       |            2 | username    | ASC       |   f     |    t      |   t
(8 rows)

        table_name        |                                 create_statement
--------------------------+-----------------------------------------------------------------------------------
  movr.abbey_schema.users | CREATE TABLE abbey_schema.users (
                          |     first_name STRING NOT NULL,
                          |     last_name STRING NOT NULL,
                          |     email STRING NOT NULL,
                          |     username STRING NOT NULL,
                          |     CONSTRAINT "primary" PRIMARY KEY (username ASC, email ASC),
                          |     UNIQUE INDEX users_first_name_last_name_key (first_name ASC, last_name ASC),
                          |     UNIQUE INDEX users_email_key (email ASC),
                          |     FAMILY "primary" (first_name, last_name, email, username)
                          | )
(1 row)
```

Note that `users_first_name_last_name_key` is a `UNIQUE` index, which means that it implies a dependent, `UNIQUE` constraint. To [drop indexes with dependencies](/docs/v25.2/drop-index#remove-an-index-and-dependent-objects-with-cascade), you can use the `CASCADE` keyword.

Create a new file, and add the `DROP` statement:

```shell
$ touch drop_unique_users_idx.sql
```

<Note>
After creation, the notation for referring to indexes in CockroachDB is `[table_name]@[index_name]`.
</Note>

```text
DROP INDEX movr.abbey_schema.users@users_first_name_last_name_key CASCADE;
```

To drop the index, execute the file:

```shell
$ cockroach sql \
--certs-dir={certs-directory} \
--user=abbey \
--database=movr \
-f drop_unique_users_idx.sql
```

```shell
$ cockroach sql \
--certs-dir={certs-directory} \
--user=abbey \
--database=movr \
--execute="SHOW INDEXES FROM movr.abbey_schema.users;"
```

```text
  table_name |   index_name    | non_unique | seq_in_index | column_name | direction | storing | implicit  | visible
-------------+-----------------+------------+--------------+-------------+-----------+---------+-----------+---------
  users      | users_pkey      |    f       |            1 | username    | ASC       |   f     |   f       |   t
  users      | users_pkey      |    f       |            2 | email       | ASC       |   f     |   f       |   t
  users      | users_email_key |    f       |            1 | email       | ASC       |   f     |   f       |   t
  users      | users_email_key |    f       |            2 | username    | ASC       |   f     |   t       |   t
(4 rows)
```
