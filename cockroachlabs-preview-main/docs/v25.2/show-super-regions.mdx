The `SHOW SUPER REGIONS` [statement](/docs/v25.2/sql-statements) lists the [super regions](/docs/v25.2/multiregion-overview#super-regions) for a multi-region cluster.

<Note>
**This feature is in [preview](/docs/v25.2/cockroachdb-feature-availability)** and subject to change. To share feedback and/or issues, contact [Support](https://support.cockroachlabs.com/hc/en-us).
</Note>

## Synopsis

<ShowRegions5 />

## Required privileges

To view the super regions in a database, the user must have one of the following:

- Membership to the [`admin`](/docs/v25.2/security-reference/authorization#admin-role) role for the cluster.
- Either [ownership](/docs/v25.2/security-reference/authorization#object-ownership) or the [`CREATE` privilege](/docs/v25.2/security-reference/authorization#supported-privileges) for the database.

## Parameters

<table><thead><tr><th>Parameter</th><th>Description</th></tr></thead><tbody><tr><td><code>FROM DATABASE database&#95;name</code></td><td>Show all <a href="/docs/v25.2/multiregion-overview#super-regions">super regions</a> for the specified database.</td></tr></tbody></table>

## Response

`SHOW SUPER REGIONS FROM DATABASE` returns the following fields for each super region:

<table><thead><tr><th>Field</th><th>Description</th></tr></thead><tbody><tr><td><code>database&#95;name</code></td><td>The database name this super region is associated with.</td></tr><tr><td><code>super&#95;region&#95;name</code></td><td>The name of a super region associated with the database.</td></tr><tr><td><code>regions</code></td><td>The list of database regions that make up the super region.</td></tr></tbody></table>

## Considerations

To use super regions, keep the following considerations in mind:

- Your cluster must be a [multi-region cluster](/docs/v25.2/multiregion-overview).
- Super regions [must be enabled](#enable-super-regions).
- Super regions can only contain one or more [database regions](/docs/v25.2/multiregion-overview#database-regions) that have already been added with [`ALTER DATABASE... ADD REGION`](/docs/v25.2/alter-database#add-region).
- Each database region can only belong to one super region. In other words, given two super regions *A* and *B*, the set of database regions in *A* must be [disjoint](https://wikipedia.org/wiki/Disjoint_sets) from the set of database regions in *B*.
- You cannot [drop a region](/docs/v25.2/alter-database#drop-region) that is part of a super region until you either [alter the super region](/docs/v25.2/alter-database#alter-super-region) to remove it, or [drop the super region](/docs/v25.2/alter-database#drop-super-region) altogether.

## Examples

The examples in this section use the following setup.

#### Setup

Only a [cluster region](/docs/v25.2/multiregion-overview#cluster-regions) specified [at node startup](/docs/v25.2/cockroach-start#locality) can be used as a [database region](/docs/v25.2/multiregion-overview#database-regions).

To follow along with the examples in this section, start a [demo cluster](/docs/v25.2/cockroach-demo) with the [`--global` flag](/docs/v25.2/cockroach-demo#general) to simulate a multi-region cluster:

```shell
$ cockroach demo --global --nodes 9
```

To see the regions available to the databases in the cluster, use a [`SHOW REGIONS FROM CLUSTER`](/docs/v25.2/show-regions#view-the-regions-in-a-cluster) statement:

```sql
SHOW REGIONS FROM CLUSTER;
```

```text
     region    |  zones
---------------+----------
  europe-west1 | {b,c,d}
  us-east1     | {b,c,d}
  us-west1     | {a,b,c}
(3 rows)
```

#### Set up movr database regions

Execute the following statements. They will tell CockroachDB about the database's regions. This information is necessary so that CockroachDB can later move data around to optimize access to particular data from particular regions. For more information about how this works at a high level, see [Database Regions](/docs/v25.2/multiregion-overview#database-regions).

```sql
ALTER DATABASE movr PRIMARY REGION "us-east1";
ALTER DATABASE movr ADD REGION "europe-west1";
ALTER DATABASE movr ADD REGION "us-west1";
```

#### Set up movr global tables

Because the data in `promo_codes` is not updated frequently (a.k.a., "read-mostly"), and needs to be available from any region, the right table locality is [`GLOBAL`](/docs/v25.2/table-localities#global-tables).

```sql
ALTER TABLE promo_codes SET locality GLOBAL;
```

Next, alter the `user_promo_codes` table to have a foreign key into the global `promo_codes` table. This will enable fast reads of the `promo_codes.code` column from any region in the cluster.

```sql
ALTER TABLE user_promo_codes
  ADD CONSTRAINT user_promo_codes_code_fk
    FOREIGN KEY (code)
    REFERENCES promo_codes (code)
    ON UPDATE CASCADE;
```

#### Set up movr regional tables

All of the tables except `promo_codes` contain rows which are partitioned by region, and updated very frequently. For these tables, the right table locality for optimizing access to their data is [`REGIONAL BY ROW`](/docs/v25.2/table-localities#regional-by-row-tables).

Apply this table locality to the remaining tables. These statements use a `CASE` statement to put data for a given city in the right region and can take around 1 minute to complete for each table.

- `rides`

    ```sql
    ALTER TABLE rides ADD COLUMN region crdb_internal_region AS (
      CASE WHEN city = 'amsterdam' THEN 'europe-west1'
           WHEN city = 'paris' THEN 'europe-west1'
           WHEN city = 'rome' THEN 'europe-west1'
           WHEN city = 'new york' THEN 'us-east1'
           WHEN city = 'boston' THEN 'us-east1'
           WHEN city = 'washington dc' THEN 'us-east1'
           WHEN city = 'san francisco' THEN 'us-west1'
           WHEN city = 'seattle' THEN 'us-west1'
           WHEN city = 'los angeles' THEN 'us-west1'
      END
    ) STORED;
    ALTER TABLE rides ALTER COLUMN REGION SET NOT NULL;
    ALTER TABLE rides SET LOCALITY REGIONAL BY ROW AS "region";
    ```
- `user_promo_codes`

    ```sql
    ALTER TABLE user_promo_codes ADD COLUMN region crdb_internal_region AS (
      CASE WHEN city = 'amsterdam' THEN 'europe-west1'
           WHEN city = 'paris' THEN 'europe-west1'
    ```

### Enable super regions

To enable super regions, set the `enable_super_regions` [session setting](/docs/v25.2/set-vars) to `'on'`:

```sql
SET enable_super_regions = 'on';
```

```text
SET
```

You can also set the `sql.defaults.super_regions.enabled` [cluster setting](/docs/v25.2/cluster-settings) to `true`:

```sql
SET CLUSTER SETTING  sql.defaults.super_regions.enabled = true;
```

```text
SET CLUSTER SETTING
```

### View the super regions from a database

`SHOW SUPER REGIONS FROM DATABASE` returns the [super regions](/docs/v25.2/multiregion-overview#super-regions) for the specified database.

```sql
SHOW SUPER REGIONS FROM DATABASE movr;
```

```text
  database_name | super_region_name |       regions
----------------+-------------------+----------------------
  movr          | usa               | {us-east1,us-west1}
(1 row)
```

<Note>
The preceding example shows the super region that was added in [`ADD SUPER REGION`](/docs/v25.2/alter-database#add-a-super-region-to-a-database).
</Note>
