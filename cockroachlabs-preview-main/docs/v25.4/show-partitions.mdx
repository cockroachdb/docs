Use the `SHOW PARTITIONS` [statement](/docs/v25.4/sql-statements) to view details about existing [partitions](/docs/v25.4/partitioning).

<Tip>
Most users should not need to use partitioning directly. Instead, they should use CockroachDB's built-in [multi-region capabilities](/docs/v25.4/multiregion-overview), which automatically handle geo-partitioning and other low-level details.
</Tip>

<Tip>
In testing, scripting, and other programmatic environments, we recommend querying the `crdb_internal.partitions` internal table for partition information instead of using the `SHOW PARTITIONS` statement. For more information, see [Querying partitions programmatically](/docs/v25.4/show-partitions#querying-partitions-programmatically).
</Tip>

## Required privileges

No [privileges](/docs/v25.4/security-reference/authorization#managing-privileges) are required to list partitions.

## Parameters

<table><thead><tr><th>Parameter</th><th>Description</th></tr></thead><tbody><tr><td><code>database&#95;name</code></td><td>The name of the <a href="/docs/v25.4/create-database">database</a> for which to show <a href="/docs/v25.4/partitioning">partitions</a>.</td></tr><tr><td><code>table&#95;name</code></td><td>The name of the <a href="/docs/v25.4/create-table">table</a> for which to show <a href="/docs/v25.4/partitioning">partitions</a>.</td></tr><tr><td><code>table&#95;index&#95;name</code></td><td>The name of the <a href="/docs/v25.4/create-index">index</a> for which to show <a href="/docs/v25.4/partitioning">partitions</a>.</td></tr></tbody></table>

## Response

The following fields are returned for each partition:

<table><thead><tr><th>Field</th><th>Description</th></tr></thead><tbody><tr><td><code>database&#95;name</code></td><td>The name of the database that contains the partition.</td></tr><tr><td><code>table&#95;name</code></td><td>The name of the table that contains the partition.</td></tr><tr><td><code>partition&#95;name</code></td><td>The name of the partition.</td></tr><tr><td><code>parent&#95;partition</code></td><td>The name of the parent partition, if the partition is a <a href="/docs/v25.4/partitioning#define-subpartitions-on-a-table">subpartition</a>.</td></tr><tr><td><code>column&#95;names</code></td><td>The names of the columns in the partition definition expression.</td></tr><tr><td><code>index&#95;name</code></td><td>The name of the index for the partition.</td></tr><tr><td><code>partition&#95;value</code></td><td>The value that defines the partition.</td></tr><tr><td><code>zone&#95;constraints</code></td><td>The <a href="/docs/v25.4/configure-replication-zones">zone constraints</a>, if replication zones are configured for the partition.</td></tr></tbody></table>

## Examples

#### Setup

The following examples use MovR, a fictional vehicle-sharing application, to demonstrate CockroachDB SQL statements. For more information about the MovR example application and dataset, see [MovR: A Global Vehicle-sharing App](/docs/v25.4/movr).

To follow along, run [`cockroach demo`](/docs/v25.4/cockroach-demo) with the `--geo-partitioned-replicas` flag. This command opens an interactive SQL shell to a temporary, 9-node in-memory cluster with the `movr` database.

```shell
$ cockroach demo --geo-partitioned-replicas
```

The `movr` database in this example is pre-partitioned. For information about partitioning tables, see [Define Table Partitions](/docs/v25.4/partitioning) or [`ALTER TABLE... PARTITION BY`](/docs/v25.4/alter-table#partition-by).

## Querying partitions programmatically

The `crdb_internal.partitions` internal table contains information about the partitions in your database. In testing, scripting, and other programmatic environments, we recommend querying this table for partition information instead of using the `SHOW PARTITIONS` statement. For example, to get all `us_west` partitions of in your database, you can run the following query:

```sql
> SELECT * FROM crdb_internal.partitions WHERE name='us_west';
```

Other internal tables, like `crdb_internal.tables`, include information that could be useful in conjunction with `crdb_internal.partitions`.

For example, if you want the output for your partitions to include the name of the table and database, you can perform a join of the two tables:

```sql
> SELECT
  partitions.name AS partition_name, column_names, list_value, tables.name AS table_name, database_name
  FROM crdb_internal.partitions JOIN crdb_internal.tables ON partitions.table_id=tables.table_id
  WHERE tables.name='users';
```

```text
  partition_name | column_names |                   list_value                    | table_name | database_name
+----------------+--------------+-------------------------------------------------+------------+---------------+
  us_west        | city         | ('seattle'), ('san francisco'), ('los angeles') | users      | movr
  us_east        | city         | ('new york'), ('boston'), ('washington dc')     | users      | movr
  europe_west    | city         | ('amsterdam'), ('paris'), ('rome')              | users      | movr
(3 rows)
```
