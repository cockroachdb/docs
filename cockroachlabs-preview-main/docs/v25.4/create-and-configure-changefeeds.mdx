Changefeeds offer different levels of configurability. Changefeeds emitting to a sink allow for active changefeed jobs to be [paused](#pause), [resumed](#resume), and [canceled](#cancel). Sinkless changefeeds stream changes directly to the SQL session.

This page describes:

- [Guidelines](#before-you-create-a-changefeed) to consider before creating a changefeed.
- [Reference examples](#configure-a-changefeed) for creating and managing a changefeed.

## Before you create a changefeed

1. Enable rangefeeds on CockroachDB Advanced and CockroachDB self-hosted. Refer to [Enable rangefeeds](#enable-rangefeeds) for instructions.
2. Decide on whether you will run a changefeed that emits to a sink or a sinkless changefeed. Refer to the [Overview](/docs/v25.4/change-data-capture-overview) page for a comparative capability table.
3. Plan the number of changefeeds versus the number of tables to include in a single changefeed for your cluster. We recommend limiting the number of changefeeds per cluster to 80. Refer to [System resources and running changefeeds](/docs/v25.4/changefeed-best-practices#maintain-system-resources-and-running-changefeeds) and [Recommendations for the number of target tables](/docs/v25.4/changefeed-best-practices#plan-the-number-of-watched-tables-for-a-single-changefeed).
    - If you're using a CockroachDB Cloud cluster, refer to [Understand CockroachDB Cloud Costs](/docs/cockroachcloud/costs) for detail on how CDC is billed monthly based on usage.
4. Consider whether your [changefeed use case](#create) would be better served by [change data capture queries](/docs/v25.4/cdc-queries) that can filter data on a single table. CDC queries can improve the efficiency of changefeeds because the job will not need to encode as much change data.
5. Read the following:
    - The [Changefeed Best Practices](/docs/v25.4/changefeed-best-practices) reference for details on planning changefeeds, monitoring basics, and schema changes.
    - The [Considerations](#considerations) section that provides information on changefeed interactions that could affect how you configure or run your changefeed.

### Enable rangefeeds

Changefeeds connect to a long-lived request called a *rangefeed*, which pushes changes as they happen. This reduces the latency of row changes, as well as reduces transaction restarts on tables being watched by a changefeed for some workloads.

**Rangefeeds must be enabled for a changefeed to work.** To [enable the cluster setting](/docs/v25.4/set-cluster-setting):

```sql
SET CLUSTER SETTING kv.rangefeed.enabled = true;
```

Any created changefeeds will error until this setting is enabled. If you are working on a CockroachDB Basic or Standard cluster, the `kv.rangefeed.enabled` cluster setting is enabled by default.

Enabling rangefeeds has a small performance cost (about a 5–10% increase in write latencies), whether or not the rangefeed is being used in a changefeed. When `kv.rangefeed.enabled` is set to `true`, a small portion of the latency cost is caused by additional write event information that is sent to the [Raft log](/docs/v25.4/architecture/replication-layer#raft-logs) and for [replication](/docs/v25.4/architecture/replication-layer). The remainder of the latency cost is incurred once a changefeed is running; the write event information is reconstructed and sent to an active rangefeed, which will push the event to the changefeed.

For further detail on performance-related configuration, refer to the [Advanced Changefeed Confguration](/docs/v25.4/advanced-changefeed-configuration) page.

<Note>
[`MuxRangefeed`](/docs/v25.4/advanced-changefeed-configuration#mux-rangefeeds) is a subsystem that improves the performance of rangefeeds with scale, which is enabled by default in v24.1 and later versions.
</Note>

### Considerations

- If you require [`resolved`](/docs/v25.4/create-changefeed#resolved) message frequency under `30s`, then you **must** set the [`min_checkpoint_frequency`](/docs/v25.4/create-changefeed#min-checkpoint-frequency) option to at least the desired `resolved` frequency.
- Many DDL queries (including [`TRUNCATE`](/docs/v25.4/truncate), [`DROP TABLE`](/docs/v25.4/drop-table), and queries that add a column family) will cause errors on a changefeed watching the affected tables. You will need to [start a new changefeed](/docs/v25.4/create-changefeed#start-a-new-changefeed-where-another-ended). If a table is truncated that a changefeed with `on_error='pause'` is watching, you will also need to start a new changefeed. Refer to the change data capture [Known Limitations](#known-limitations) for more detail.
- Partial or intermittent sink unavailability may impact changefeed stability. If a sink is unavailable, messages can't send, which means that a changefeed's high-water mark timestamp is at risk of falling behind the cluster's [garbage collection window](/docs/v25.4/configure-replication-zones#replication-zone-variables). Throughput and latency can be affected once the sink is available again. However, [ordering guarantees](/docs/v25.4/changefeed-messages#ordering-and-delivery-guarantees) will still hold for as long as a changefeed [remains active](/docs/v25.4/monitor-and-debug-changefeeds#monitor-a-changefeed).
- When an [`IMPORT INTO`](/docs/v25.4/import-into) statement is run, any current changefeed jobs targeting that table will fail.
- After you [restore from a full-cluster backup](/docs/v25.4/restore#full-cluster), changefeed jobs will **not** resume on the new cluster. It is necessary to manually create the changefeeds following the full-cluster restore.
- As of v22.1, changefeeds filter out [`VIRTUAL` computed columns](/docs/v25.4/computed-columns) from events by default. This is a [backward-incompatible change](/docs/releases/v22.1#v22-1-0-backward-incompatible-changes). To maintain the changefeed behavior in previous versions where [`NULL`](/docs/v25.4/null-handling) values are emitted for virtual computed columns, see the [`virtual_columns`](/docs/v25.4/create-changefeed#virtual-columns) option for more detail.

## Configure a changefeed

A changefeed streams row-level changes in a [configurable format](/docs/v25.4/changefeed-messages) to one of the following sinks:

- [Amazon MSK](/docs/v25.4/changefeed-sinks#amazon-msk)
- [Apache Pulsar](/docs/v25.4/changefeed-sinks#apache-pulsar) (in Preview)
- [Azure Event Hubs](/docs/v25.4/changefeed-sinks#azure-event-hubs)
- [Cloud Storage](/docs/v25.4/changefeed-sinks#cloud-storage-sink) / HTTP
- [Confluent Cloud](/docs/v25.4/changefeed-sinks#confluent-cloud)
- [Google Cloud Pub/Sub](/docs/v25.4/changefeed-sinks#google-cloud-pub-sub)
- [Kafka](/docs/v25.4/changefeed-sinks#kafka)
- [Webhook](/docs/v25.4/changefeed-sinks#webhook-sink)

You can [create](#create), [pause](#pause), [resume](#resume), and [cancel](#cancel) a changefeed emitting messages to a sink. For a step-by-step example connecting to a specific sink, see the [Changefeed Examples](/docs/v25.4/changefeed-examples) page.

### Create

To create a changefeed:

```sql
CREATE CHANGEFEED FOR TABLE table_name, table_name2 INTO '{scheme}://{sink_host}:{port}?{query_parameters}';
```

<Note>
Parameters should always be URI-encoded before they are included the changefeed's URI, as they often contain special characters. Use Javascript's [encodeURIComponent](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Global_Objects/encodeURIComponent) function or Go language's [url.QueryEscape](https://golang.org/pkg/net/url#QueryEscape) function to URI-encode the parameters. Other languages provide similar functions to URI-encode special characters.
</Note>

For more information, see [`CREATE CHANGEFEED`](/docs/v25.4/create-changefeed).

### Pause

To pause a changefeed:

```sql
PAUSE JOB job_id;
```

For more information, refer to [`PAUSE JOB`](/docs/v25.4/pause-job).

### Resume

To resume a paused changefeed:

```sql
RESUME JOB job_id;
```

For more information, refer to [`RESUME JOB`](/docs/v25.4/resume-job).

### Cancel

To cancel a changefeed:

```sql
CANCEL JOB job_id;
```

For more information, refer to [`CANCEL JOB`](/docs/v25.4/cancel-job).

### Modify a changefeed

To modify a changefeed, [pause](/docs/v25.4/create-and-configure-changefeeds#pause) the job and then use:

```sql
ALTER CHANGEFEED job_id {ADD table DROP table SET option UNSET option};
```

You can add new table targets, remove them, set new [changefeed options](/docs/v25.4/create-changefeed#options), and unset them.

For more information, see [`ALTER CHANGEFEED`](/docs/v25.4/alter-changefeed).

### Configuring all changefeeds

It is useful to be able to pause all running changefeeds during troubleshooting, testing, or when a decrease in CPU load is needed.

To pause all running changefeeds:

```sql
PAUSE JOBS (WITH x AS (SHOW CHANGEFEED JOBS) SELECT job_id FROM x WHERE status = ('running'));
```

This will change the status for each of the running changefeeds to `paused`, which can be verified with [`SHOW CHANGEFEED JOBS`](/docs/v25.4/show-jobs#show-changefeed-jobs).

To resume all running changefeeds:

```sql
RESUME JOBS (WITH x AS (SHOW CHANGEFEED JOBS) SELECT job_id FROM x WHERE status = ('paused'));
```

This will resume the changefeeds and update the status for each of the changefeeds to `running`.

## Create a sinkless changefeed

When you create a changefeed **without** specifying a sink (a sinkless changefeed), CockroachDB sends the changefeed events to the SQL client indefinitely until the underlying connection is closed or the changefeed is canceled:

```sql
CREATE CHANGEFEED FOR TABLE table_name, table_name2;
```

Consider the following regarding the [display format](/docs/v25.4/cockroach-sql#sql-flag-format) in your SQL client:

- If you do not define a display format, the CockroachDB SQL client will automatically use `ndjson` format.
- If you specify a display format, the client will use that format (e.g., `--format=csv` ).
- If you set the client display format to `ndjson` and set the changefeed [`format`](/docs/v25.4/create-changefeed#format) to `csv`, you'll receive JSON format with CSV nested inside.
- If you set the client display format to `csv` and set the changefeed [`format`](/docs/v25.4/create-changefeed#format) to `json`, you'll receive a comma-separated list of JSON values.

For more information, see [`CREATE CHANGEFEED`](/docs/v25.4/create-changefeed#create-a-sinkless-changefeed).

## Known limitations

- Changefeed target options are limited to tables and [column families](/docs/v25.4/changefeeds-on-tables-with-column-families). [#73435](https://github.com/cockroachdb/cockroach/issues/73435)
- [VPC Peering](/docs/cockroachcloud/network-authorization#vpc-peering) and [AWS PrivateLink](/docs/cockroachcloud/network-authorization#aws-privatelink) in CockroachDB Advanced clusters do **not** support connecting to a [Kafka](/docs/v25.4/changefeed-sinks#kafka) sink's internal IP addresses for [changefeeds](/docs/v25.4/change-data-capture-overview). To connect to a Kafka sink from CockroachDB Advanced, it is necessary to expose the Kafka cluster's external IP address and open ports with firewall rules to allow access from a CockroachDB Advanced cluster.
- Webhook sinks only support HTTPS. Use the [`insecure_tls_skip_verify`](/docs/v25.4/create-changefeed#insecure-tls-skip-verify) parameter when testing to disable certificate verification; however, this still requires HTTPS and certificates. [#73431](https://github.com/cockroachdb/cockroach/issues/73431)
- Formats for changefeed messages are not supported by all changefeed sinks. Refer to the [Changefeed Sinks](/docs/v25.4/changefeed-sinks) page for details on compatible formats with each sink and the [`format`](/docs/v25.4/create-changefeed) option to specify a changefeed message format. [#73432](https://github.com/cockroachdb/cockroach/issues/73432)
- Using the [`split_column_families`](/docs/v25.4/create-changefeed#split-column-families) and [`resolved`](/docs/v25.4/create-changefeed#resolved) options on the same changefeed will cause an error when using the following [sinks](/docs/v25.4/changefeed-sinks): Kafka and Google Cloud Pub/Sub. Instead, use the individual `FAMILY` keyword to specify column families when creating a changefeed. [#79452](https://github.com/cockroachdb/cockroach/issues/79452)
- Changefeed types are not fully integrated with [user-defined composite types](/docs/v25.4/create-type). Running changefeeds with user-defined composite types is in [Preview](/docs/v25.4/cockroachdb-feature-availability#feature-availability-phases). Certain changefeed types do not support user-defined composite types. Refer to the change data capture [Known Limitations](/docs/v25.4/create-and-configure-changefeeds#known-limitations) for more detail. The following limitations apply:
    - A changefeed in [Avro format](/docs/v25.4/changefeed-messages#avro) will not be able to serialize [user-defined composite (tuple) types](/docs/v25.4/create-type). [#102903](https://github.com/cockroachdb/cockroach/issues/102903)
    - A changefeed emitting [CSV](/docs/v25.4/changefeed-messages#csv) will include `AS` labels in the message format when the changefeed serializes a [user-defined composite type](/docs/v25.4/create-type). [#102905](https://github.com/cockroachdb/cockroach/issues/102905)
- [`ALTER CHANGEFEED`](/docs/v25.4/alter-changefeed) is not fully supported with changefeeds that use [CDC queries](/docs/v25.4/cdc-queries). You can alter the options that a changefeed uses, but you cannot alter the changefeed target tables. [#83033](https://github.com/cockroachdb/cockroach/issues/83033)
- Creating a changefeed with [CDC queries](/docs/v25.4/cdc-queries) on tables with more than one [column family](/docs/v25.4/changefeeds-on-tables-with-column-families) is not supported. [#127761](https://github.com/cockroachdb/cockroach/issues/127761)
- When you create a changefeed on a table with more than one [column family](/docs/v25.4/changefeeds-on-tables-with-column-families), the changefeed will emit messages per column family in separate streams. As a result, [changefeed messages](/docs/v25.4/changefeed-messages) for different column families will arrive at the [sink](/docs/v25.4/changefeed-sinks) under separate topics. [#127736](https://github.com/cockroachdb/cockroach/issues/127736)
