The `CREATE USER` [statement](/docs/v25.4/sql-statements) creates SQL users, which let you control [privileges](/docs/v25.4/security-reference/authorization#managing-privileges) on your databases and tables.

You can use the keywords `ROLE` and `USER` interchangeably. `CREATE USER` is equivalent to [`CREATE ROLE`](/docs/v25.4/create-role), with one exception: `CREATE ROLE` sets the `NOLOGIN` [role option](#role-options), which prevents the new role from being used to log in to the database. You can use `CREATE ROLE` and specify the `LOGIN` [role option](#role-options) to achieve the same result as `CREATE USER`.

<Note>
The `CREATE USER` statement performs a schema change. For more information about how online schema changes work in CockroachDB, see [Online Schema Changes](/docs/v25.4/online-schema-changes).
</Note>

## Considerations

- After creating users, you must [grant them privileges to databases and tables](/docs/v25.4/grant).
- All users belong to the `public` role, to which you can [grant](/docs/v25.4/grant) and [revoke](/docs/v25.4/revoke) privileges.
- On secure clusters, you must [create client certificates for users](/docs/v25.4/cockroach-cert#create-the-certificate-and-key-pair-for-a-client) and users must [authenticate their access to the cluster](#user-authentication).
- <InlineImage alt="Megaphone" src="/images/common/icon-megaphone.png" /> New in v25.4: The `CREATE USER` statement waits for full-cluster visibility of the new user metadata. It is not blocked by long-running transactions that may have accessed older metadata, and it does not create a [background job](/docs/v25.4/show-jobs).

## Required privileges

To create other users, the user must be a member of the `admin` role or have the [`CREATEROLE`](#create-a-user-that-can-create-other-users-and-manage-authentication-methods-for-the-new-users) parameter set.

### User names

- Are case-insensitive.
- Must start with a letter or underscore.
- Must contain only letters, numbers, periods, or underscores.
- Must be between 1 and 63 characters.
- Cannot be `none`.
- Cannot start with `pg_` or `crdb_internal`. Object names with these prefixes are reserved for [system catalogs](/docs/v25.4/system-catalogs).
- User and role names share the same namespace and must be unique.

## User authentication

Secure clusters require users to authenticate their access to databases and tables. CockroachDB offers three methods for this:

- [Client certificate and key authentication](/docs/v25.4/authentication#client-authentication), which is available to all users. To ensure the highest level of security, we recommend only using client certificate and key authentication.
- [Password authentication](#create-a-user-with-a-password), which is available to users and roles who you've created passwords for. To create a user with a password, use the `WITH PASSWORD` clause of `CREATE USER`. To add a password to an existing user, use the [`ALTER USER`](/docs/v25.4/alter-user) statement.

    Users can use passwords to authenticate without supplying client certificates and keys; however, we recommend using certificate-based authentication whenever possible.

    Password creation is supported only in secure clusters.
- [GSSAPI authentication](/docs/v25.4/gssapi_authentication).

## Examples

To run the following examples, [start a secure single-node cluster](/docs/v25.4/cockroach-start-single-node) and use the built-in SQL shell:

```shell
$ cockroach sql --certs-dir=certs
```

```sql
> SHOW USERS;
```

```text
username | options | member_of | estimated_last_login_time
---------+---------+-----------+------------------------------
admin    | {}      | {}        | NULL
root     | {}      | {admin}   | NULL
(2 rows)
```

<Note>
The following statements are run by the `root` user that is a member of the `admin` role and has `ALL` privileges.
</Note>

### Create a user

Usernames are case-insensitive; must start with a letter or underscore; must contain only letters, numbers, periods, or underscores; and must be between 1 and 63 characters.

```sql
root@:26257/defaultdb> CREATE USER no_options;
```

```sql
root@:26257/defaultdb> SHOW USERS;
```

```text
 username  | options | member_of | estimated_last_login_time
-----------+---------+-----------+------------------------------
admin      | {}      | {}        | NULL
no_options | {}      | {}        | NULL
root       | {}      | {admin}   | NULL
(3 rows)
```

After creating users, you must:

- [Grant them privileges to databases](/docs/v25.4/grant).
- For secure clusters, you must also [create their client certificates](/docs/v25.4/cockroach-cert#create-the-certificate-and-key-pair-for-a-client).

### Create a user with a password

```sql
root@:26257/defaultdb> CREATE USER with_password WITH LOGIN PASSWORD '$tr0nGpassW0rD' VALID UNTIL '2021-10-10';
```

```sql
root@:26257/defaultdb> SHOW USERS;
```

```text
  username    |                  options                  | member_of | estimated_last_login_time
--------------+-------------------------------------------+-----------+------------------------------

root          | {}                                        | {admin}   | NULL
with_password | {VALID UNTIL=2025-10-10 00:00:00+00:00}   | {}        | NULL
(4 rows)
```

### Prevent a user from using password authentication

The following statement prevents the user from using password authentication and mandates certificate-based client authentication:

```sql
root@:26257/defaultdb> CREATE USER no_password WITH PASSWORD NULL;
```

```sql
root@:26257/defaultdb> SHOW USERS;
```
