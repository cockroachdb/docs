### `ADD REGION`

`ALTER DATABASE.. ADD REGION` adds a [region](/docs/v25.4/multiregion-overview#database-regions) to a [multi-region database](/docs/v25.4/multiregion-overview). While CockroachDB processes an index modification or changing a table to or from a [`REGIONAL BY ROW` table](/docs/v25.4/table-localities#regional-by-row-tables), attempting to drop a region from the database containing that `REGIONAL BY ROW` table will produce an error. Similarly, while this statement is running, all index modifications and locality changes on [`REGIONAL BY ROW`](/docs/v25.4/table-localities#regional-by-row-tables) tables will be blocked.

<Warning>
In order to add a region with `ADD REGION`, you must first set a primary database region with [`SET PRIMARY REGION`](#set-primary-region), or at [database creation](/docs/v25.4/create-database). For an example showing how to add a primary region with `ALTER DATABASE`, see [Set the primary region](#set-the-primary-region).
</Warning>

For examples, see [Manage regions](#manage-regions).

#### Required privileges

To add a region to a database, the user must have one of the following:

- Membership to the [`admin`](/docs/v25.4/security-reference/authorization#admin-role) role for the cluster.
- Either [ownership](/docs/v25.4/security-reference/authorization#object-ownership) or the [`CREATE` privilege](/docs/v25.4/security-reference/authorization#supported-privileges) for the database and all [`REGIONAL BY ROW`](/docs/v25.4/table-localities#regional-by-row-tables) tables in the database.

Allowed values include any region present in <a href="/docs/v25.4/show-regions"><code>SHOW REGIONS FROM CLUSTER</code></a>.</td></tr></tbody></table>

### `ADD SUPER REGION`

`ALTER DATABASE.. ADD SUPER REGION` adds a [super region](/docs/v25.4/multiregion-overview#super-regions) to a [multi-region database](/docs/v25.4/multiregion-overview).

<Note>
**This feature is in [preview](/docs/v25.4/cockroachdb-feature-availability)** and subject to change. To share feedback and/or issues, contact [Support](https://support.cockroachlabs.com/hc/en-us).
</Note>

To use super regions, keep the following considerations in mind:

- Your cluster must be a [multi-region cluster](/docs/v25.4/multiregion-overview).
- Super regions [must be enabled](/docs/v25.4/alter-database#enable-super-regions).
- Super regions can only contain one or more [database regions](/docs/v25.4/multiregion-overview#database-regions) that have already been added with [`ALTER DATABASE... ADD REGION`](/docs/v25.4/alter-database#add-region).
- Each database region can only belong to one super region. In other words, given two super regions *A* and *B*, the set of database regions in *A* must be [disjoint](https://wikipedia.org/wiki/Disjoint_sets) from the set of database regions in *B*.
- You cannot [drop a region](/docs/v25.4/alter-database#drop-region) that is part of a super region until you either [alter the super region](/docs/v25.4/alter-database#alter-super-region) to remove it, or [drop the super region](/docs/v25.4/alter-database#drop-super-region) altogether.

For examples, see [Manage super regions](#manage-super-regions).

#### Required privileges

To add a super region to a database, the user must have one of the following:

- Membership to the [`admin`](/docs/v25.4/security-reference/authorization#admin-role) role for the cluster.
- Either [ownership](/docs/v25.4/security-reference/authorization#object-ownership) or the [`CREATE` privilege](/docs/v25.4/security-reference/authorization#supported-privileges) for the database.

### `ALTER LOCALITY`

`ALTER DATABASE.. ALTER LOCALITY` provides a customization tool for advanced users to persistently modify the configuration generated by the standard [multi-region SQL abstractions](/docs/v25.4/multiregion-overview) on a per-region basis.

The feature this statement enables is also known as *Zone Config Extensions*. For more information about how it works, and the benefits over using the low-level [`CONFIGURE ZONE`](#configure-zone) statement, see [Zone Config Extensions](/docs/v25.4/zone-config-extensions).

For examples, see [Use Zone Config Extensions](#use-zone-config-extensions).

#### Required privileges

The user must be a member of the [`admin`](/docs/v25.4/security-reference/authorization#roles) or [owner](/docs/v25.4/security-reference/authorization#object-ownership) roles, or have the [`CREATE` privilege](/docs/v25.4/security-reference/authorization#supported-privileges) on the database.

### `CONFIGURE ZONE`

`ALTER DATABASE... CONFIGURE ZONE` is used to add, modify, reset, or remove replication zones for a database. To view details about existing replication zones, use [`SHOW ZONE CONFIGURATIONS`](/docs/v25.4/show-zone-configurations). For more information about replication zones, see [Replication Controls](/docs/v25.4/configure-replication-zones).

<Warning>
If you directly change a database's zone configuration with `ALTER DATABASE... CONFIGURE ZONE`, CockroachDB will block all [`ALTER DATABASE... SET PRIMARY REGION`](#set-primary-region) statements on the database.
</Warning>

<Warning>
Cockroach Labs [does not recommend modifying zone configurations manually](/docs/v25.4/configure-replication-zones#why-manual-zone-config-management-is-not-recommended).

Most users should use [multi-region SQL statements](/docs/v25.4/multiregion-overview) instead. If additional control is needed, use [Zone Config Extensions](/docs/v25.4/zone-config-extensions) to augment the multi-region SQL statements.
</Warning>

You can use *replication zones* to control the number and location of replicas for specific sets of data, both when replicas are first added and when they are rebalanced to maintain cluster equilibrium.

For examples, see [Replication Controls](#configure-replication-zones).

#### Required privileges

The user must be a member of the [`admin` role](/docs/v25.4/security-reference/authorization#admin-role) or have been granted [`CREATE`](/docs/v25.4/security-reference/authorization#supported-privileges) or [`ZONECONFIG`](/docs/v25.4/security-reference/authorization#supported-privileges) privileges. To configure [`system` objects](/docs/v25.4/configure-replication-zones#for-system-data), the user must be a member of the `admin` role.

### `DROP REGION`

`ALTER DATABASE.. DROP REGION` drops a [region](/docs/v25.4/multiregion-overview#database-regions) from a [multi-region database](/docs/v25.4/multiregion-overview). While CockroachDB processes an index modification or changing a table to or from a [`REGIONAL BY ROW` table](/docs/v25.4/table-localities#regional-by-row-tables), attempting to drop a region from the database containing that `REGIONAL BY ROW` table will produce an error. Similarly, while this statement is running, all index modifications and locality changes on [`REGIONAL BY ROW`](/docs/v25.4/table-localities#regional-by-row-tables) tables will be blocked.

You cannot drop a region from a multi-region database if:

- The region is [set as the primary region](#set-primary-region), unless it is the last remaining region.
- The region is [set as the secondary region](#set-secondary-region). To drop the region, you must first unset the secondary region using [`DROP SECONDARY REGION`](#drop-secondary-region).
- The database uses the [`REGION` survival goal](/docs/v25.4/multiregion-survival-goals#survive-region-failures) and there are only three regions configured on the database.

For examples, see [Manage regions](#manage-regions).

#### Required privileges

To drop a region from a database, the user must have one of the following:

- Membership to the [`admin`](/docs/v25.4/security-reference/authorization#roles) role for the cluster.
- Membership to the [owner](/docs/v25.4/security-reference/authorization#object-ownership) role, or the [`CREATE` privilege](/docs/v25.4/security-reference/authorization#supported-privileges), for the database and all [`REGIONAL BY ROW`](/docs/v25.4/table-localities#regional-by-row-tables) tables in the database.

### `PLACEMENT`

`ALTER DATABASE... PLACEMENT` is used to set the replica placement policy (`DEFAULT` or `RESTRICTED`) for a [multi-region database](/docs/v25.4/multiregion-overview)'s [regional tables](/docs/v25.4/regional-tables). [Regional tables](/docs/v25.4/regional-tables) are those with [`REGIONAL BY ROW`](/docs/v25.4/table-localities#regional-by-row-tables) or [`REGIONAL BY TABLE`](/docs/v25.4/table-localities#regional-tables) localities.

`ALTER DATABASE... PLACEMENT RESTRICTED` constrains the voting and non-voting replica placement for the database's tables to the [home regions](/docs/v25.4/alter-table#crdb_region) associated with those tables. This is a way of opting out of [non-voting replicas](/docs/v25.4/architecture/replication-layer#non-voting-replicas) for [regional tables](/docs/v25.4/regional-tables) to accomplish one or more of the following goals:

- Implement a [data domiciling](/docs/v25.4/data-domiciling) strategy.
- Reduce the amount of data stored on the cluster.
- Reduce the overhead of replicating data across a large number of regions (e.g., 10 or more) for databases with heavier write loads.

Note that `ALTER DATABASE... PLACEMENT RESTRICTED` does not allow you to opt out of placing [non-voting replicas](/docs/v25.4/architecture/replication-layer#non-voting-replicas) entirely. For example, [`GLOBAL` tables](/docs/v25.4/global-tables) in the database will remain unaffected by this statement. `GLOBAL` tables are designed to have replicas placed across all available [cluster regions](/docs/v25.4/multiregion-overview#cluster-regions) to ensure fast local reads.

For examples, see [Set replica placement policy](#set-replica-placement-policy).

#### Required privileges

To use this statement, the user must have one of the following:

- Membership to the [`admin`](/docs/v25.4/security-reference/authorization#roles) role for the cluster.
- [Ownership](/docs/v25.4/security-reference/authorization#object-ownership) or the [`CREATE` privilege](/docs/v25.4/security-reference/authorization#supported-privileges) for the database and all tables in the database.

#### Parameters

<table><thead><tr><th>Parameter</th><th>Description</th></tr></thead><tbody><tr><td><code>DEFAULT</code></td><td>Set the replica placement policy for <a href="/docs/v25.4/regional-tables">regional tables</a> to <code>DEFAULT</code>.</td></tr><tr><td><code>RESTRICTED</code></td><td>Set the replica placement policy for <a href="/docs/v25.4/regional-tables">regional tables</a> to <code>RESTRICTED</code>.</td></tr></tbody></table>

For usage, see [Synopsis](#synopsis).

The replica placement policies work as follows:

- `DEFAULT` (**Default**): If the replica placement policy [is set to `DEFAULT`](#set-the-replica-placement-policy-to-default), CockroachDB will use its default replica placement settings, which mean that:
    - Data will be placed in as many regions as necessary to ensure your [database survival goals](/docs/v25.4/multiregion-overview#survival-goals) are met.
    - You can get fast stale reads from all [database regions](/docs/v25.4/multiregion-overview#database-regions).
- `RESTRICTED`: If the replica placement policy [is set to `RESTRICTED`](#set-the-replica-placement-policy-to-restricted), CockroachDB will constrain replica placement to only those regions where the table has voting replicas (that is, replicas which participate in the [Raft quorum](/docs/v25.4/architecture/replication-layer#raft)). In practice, this means that voting replicas for the table will be constrained to the table's [home region](/docs/v25.4/multiregion-overview#table-localities). Specifically, for [`REGIONAL BY TABLE`](/docs/v25.4/table-localities#regional-tables) tables, it will only place replicas in the defined region (or the database's [primary region](#set-primary-region)); for [`REGIONAL BY ROW`](/docs/v25.4/table-localities#regional-by-row-tables) tables, it will only place replicas for each underlying [partition](/docs/v25.4/partitioning) in the partition's specified region. Finally, note that:
    - Regional tables with this placement setting will no longer provide "fast stale reads" from other (non-home) regions, since fast stale reads rely on the presence of non-voting replicas.
    - The `RESTRICTED` replica placement policy is only available for databases with the [`ZONE` survival goal](/docs/v25.4/multiregion-survival-goals#survive-zone-failures).
    - This setting does not affect how [`GLOBAL` tables](/docs/v25.4/global-tables) work; they will still place replicas in all [database regions](/docs/v25.4/multiregion-overview#database-regions).

#### Session variable precedence

When a [session](/docs/v25.4/show-sessions) starts, CockroachDB determines the initial value of each [session variable](/docs/v25.4/session-variables) by evaluating the settings in the following order (items earlier in the list take precedence over later items):

1. [Connection string parameters](/docs/v25.4/connection-parameters): A value supplied as a query parameter in the connection URL (for example, `.../movr?sslmode=disable&timezone=UTC` ).
2. [Per-role, per-database defaults](/docs/v25.4/alter-role#set-default-session-variable-values-for-a-role-in-a-specific-database): A value set by `ALTER ROLE {role_name} IN DATABASE {db_name} SET {var}={value}`.
3. [Per-role, all-database defaults](/docs/v25.4/alter-role#set-default-session-variable-values-for-a-role): A value set by `ALTER ROLE {role_name} SET {var}={value}`.
4. [All-role, per-database defaults](/docs/v25.4/alter-role#set-default-session-variable-values-for-a-specific-database): A value set by `ALTER ROLE ALL IN DATABASE {db_name} SET {var}={value}` or equivalently by `ALTER DATABASE {db_name} SET {var}={value}`.
5. [Cluster-wide defaults for all roles and all databases](/docs/v25.4/alter-role#set-default-session-variable-values-for-all-users): A value set by `ALTER ROLE ALL SET {var}={value}`.

If a session variable is not modified using any of the preceding methods, the built-in system default value is used. Note that the [`root` user](/docs/v25.4/security-reference/authorization#root-user) is exempt from settings 3–5. The `root` user is only affected by values specified in the connection string.

You can also set session variables for the duration of a single transaction by using [`SET LOCAL {var}={value}`](/docs/v25.4/set-vars#set-a-variable-for-the-duration-of-a-single-transaction).

<Note>
Changes to defaults using the preceding methods only apply to **future** sessions. This is because session variable resolution happens at session start time. To change a default value in an existing open session, set the variable explicitly with [`SET`](/docs/v25.4/set-vars).
</Note>

### `SET PRIMARY REGION`

`ALTER DATABASE.. SET PRIMARY REGION` sets the primary [region](/docs/v25.4/multiregion-overview#database-regions) of a [multi-region database](/docs/v25.4/multiregion-overview).

- Membership to the [owner](/docs/v25.4/security-reference/authorization#object-ownership) role, or the [`CREATE` privilege](/docs/v25.4/security-reference/authorization#supported-privileges), for the database and all tables in the database.

To switch primary regions to a region that has already been added to a database, the user must have membership to the [owner](/docs/v25.4/security-reference/authorization#object-ownership) role for the database, or have the [`CREATE` privilege](/docs/v25.4/security-reference/authorization#supported-privileges) on the database.

#### Parameters

For usage, see [Synopsis](#synopsis).

#### Set the primary region

Suppose you want to make the database `movr` a multi-region database.

To add the first region to the database, or to set an already-added region as the primary region, use a [`SET PRIMARY REGION`](#set-primary-region) statement:

```sql
ALTER DATABASE movr SET PRIMARY REGION "us-east1";
```

```text
ALTER DATABASE PRIMARY REGION
```

Given a cluster with multiple regions, any databases in that cluster that have not yet had their primary regions set will have their replicas spread as broadly as possible for resiliency. When a primary region is added to one of these databases:

- All tables will be [`REGIONAL BY TABLE`](/docs/v25.4/alter-table#regional-by-table) in the primary region by default.
- This means that all such tables will have all of their voting replicas and leaseholders moved to the primary region. This process is known as [rebalancing](/docs/v25.4/architecture/replication-layer#leaseholder-rebalancing).

#### Add regions to a database

To add more regions to a database that already has at least one region:

```sql
ALTER DATABASE movr ADD region "us-west1";
```

```text
ALTER DATABASE ADD REGION
```

```sql
ALTER DATABASE movr ADD region "europe-west1";
```

```text
ALTER DATABASE ADD REGION
```

To view the regions associated with a multi-region database, use a [`SHOW REGIONS FROM DATABASE`](/docs/v25.4/show-regions) statement:

```sql
SHOW REGIONS FROM DATABASE movr;
```

```text
  database |    region    | primary | secondary |  zones
-----------+--------------+---------+-----------+----------
  movr     | us-east1     |    t    |     f     | {b,c,d}
  movr     | europe-west1 |    f    |     f     | {b,c,d}
  movr     | us-west1     |    f    |     f     | {a,b,c}
(3 rows)
```

#### Set the secondary region

To set an existing [database region](/docs/v25.4/multiregion-overview#database-regions) (that is not already the [primary region](#set-the-primary-region)) as the [secondary region](/docs/v25.4/multiregion-overview#secondary-regions), use the following statement:

```sql
ALTER DATABASE movr SET SECONDARY REGION "us-west1";
```

```text
ALTER DATABASE SET SECONDARY REGION
```

Now, the `"us-west1"` region will act as the primary region if the original primary region fails.

<Warning>
[Secondary regions](/docs/v25.4/multiregion-overview#secondary-regions) are not compatible with databases containing [`REGIONAL BY ROW`](/docs/v25.4/table-localities#regional-by-row-tables) tables. CockroachDB does not prevent you from defining secondary regions on databases with regional by row tables, but the interaction of these features is not supported. Therefore, Cockroach Labs recommends that you avoid defining secondary regions on databases that use regional by row table configurations.
</Warning>

#### Unset the secondary region

To unset an existing [database region](/docs/v25.4/multiregion-overview#database-regions) as the [secondary region](/docs/v25.4/multiregion-overview#secondary-regions) on a multi-region database, use the [`DROP SECONDARY REGION`](#drop-secondary-region) statement:

```sql
ALTER DATABASE movr DROP SECONDARY REGION;
```

```text
ALTER DATABASE DROP SECONDARY REGION
```

<Note>
This statement does not drop the region from the database. To drop a region, use [`DROP REGION`](#drop-a-region-from-a-database).
</Note>

#### Change the existing primary region

To change the primary region to another region in the database, use a [`SET PRIMARY REGION`](#set-primary-region) statement.

```sql
ALTER DATABASE movr SET PRIMARY REGION "europe-west1";
```

```text
ALTER DATABASE PRIMARY REGION
```

You can only change an existing primary region to a region that has already been [added to the database](#add-regions-to-a-database). If you try to change the primary region to a region that is not already associated with a database, CockroachDB will return an error.

#### Drop a region from a database

To [drop a region](#drop-region) from a multi-region database, use a [`DROP REGION`](#drop-region) statement.

```sql
ALTER DATABASE movr DROP REGION "us-east1";
```

```text
ALTER DATABASE DROP REGION
```

You cannot drop a region from a multi-region database if:

- The region is [set as the primary region](#set-the-primary-region), unless it is the last remaining region.
- The region is [set as the secondary region](#set-the-secondary-region). To drop the region, you must first unset the secondary region using [`DROP SECONDARY REGION`](#unset-the-secondary-region).
- The database uses the [`REGION` survival goal](/docs/v25.4/multiregion-survival-goals#survive-region-failures) and there are only three regions configured on the database.

To add a super region to a multi-region database, use the `ALTER DATABASE... ADD SUPER REGION` statement:

```sql
ALTER DATABASE movr ADD SUPER REGION "usa" VALUES "us-east1", "us-west1";
```

```text
ALTER DATABASE ADD SUPER REGION
```

#### Alter a super region

This example assumes you have already added a `"usa"` super region as shown in the example [Add a super region to a database](#add-a-super-region-to-a-database). If you wanted to [drop the region](#drop-region) `us-west1`, you would first need to remove it from the super region.

To remove a region from a super region, use the `ALTER DATABASE... ALTER SUPER REGION` statement and list only the regions that should remain in the super region:

```sql
ALTER DATABASE movr ALTER SUPER REGION "usa" VALUES "us-east1";
```

```text
ALTER DATABASE ALTER SUPER REGION
```

To add a region to a super region, alter the super region as shown above to be a list of regions that includes the existing and the new regions.

#### Allow user to modify a primary region that is part of a super region

By default, you may not change the [primary region](/docs/v25.4/alter-database#set-primary-region) of a [multi-region database](/docs/v25.4/multiregion-overview) when that region is part of a super region. This is a safety setting designed to prevent you from accidentally moving the data for a [regional table](/docs/v25.4/regional-tables) that is meant to be stored in the super region out of that super region, which could break your data domiciling setup.

If you are sure about what you are doing, you can allow modifying the primary region by setting the `alter_primary_region_super_region_override` [session setting](/docs/v25.4/set-vars) to `'on'`:

```sql
SET alter_primary_region_super_region_override = 'on';
```

```text
SET
```

You can also accomplish this by setting the `sql.defaults.alter_primary_region_super_region_override.enable` [cluster setting](/docs/v25.4/cluster-settings) to `true`:

```sql
SET CLUSTER SETTING  sql.defaults.alter_primary_region_super_region_override.enable = true;
```

```text
SET CLUSTER SETTING
```

#### Drop a super region from a database

To drop a super region from a multi-region database, use a [`DROP SUPER REGION`](#drop-super-region) statement:

```sql
ALTER DATABASE movr DROP SUPER REGION "usa";
```

```text
ALTER DATABASE DROP SUPER REGION
```

Note that you cannot [drop a region](#drop-region) that is part of a super region until you either [alter the super region](#alter-super-region) to remove it, or [drop the super region](#drop-super-region) altogether.

For example, using the super region that was added in [`ADD SUPER REGION`](#add-a-super-region-to-a-database):

```sql
ALTER DATABASE movr DROP REGION "us-west1";
```


#### Implement super regions

In this example, [Zone Config Extensions](/docs/v25.4/zone-config-extensions) are used to provide an alternative implementation of [super regions](/docs/v25.4/multiregion-overview#super-regions), which are useful for [data domiciling](/docs/v25.4/data-domiciling).

For this example, you need to start a [`cockroach demo` cluster](/docs/v25.4/cockroach-demo) using the following command, which [defines the available cluster localities using the `--demo-locality` flag](/docs/v25.4/cockroach-demo#demo-locality):

```shell

```

Next, enter the following statements at the SQL prompt. These statements will:

- Make the [`movr` database](/docs/v25.4/movr) into a [multi-region database](/docs/v25.4/multiregion-overview) by [setting the primary region](/docs/v25.4/alter-database#set-primary-region).
- Set the `movr` database to [survive region failures](/docs/v25.4/multiregion-survival-goals#survive-region-failures).
- Add the remaining regions defined at cluster start time to the multi-region `movr` database (for a total of 6).

```sql
ALTER DATABASE movr ADD REGION "us-west1";
```

- [Replicas](/docs/v25.4/architecture/overview#architecture-replica) in the `us-east1` region are only stored on nodes in the US.
- Replicas in the `europe-central1` region are only stored on nodes in the EU.

```sql
    num_replicas = 7,
    constraints = '{+region=europe-central1: 3,
                    +region=europe-west1: 3,
                    +region=europe-east1: 1}',
```

#### Minimize cross-region write latency

In this example, [Zone Config Extensions](/docs/v25.4/zone-config-extensions) are used to minimize cross-region write latency in a cluster with the following characteristics:

- Three main US regions ( `us-west1`, `us-east1`, `us-central1` ).
- Two additional distant regions ( `europe-west1`, `asia-northeast1` ). Only [follower reads](/docs/v25.4/follower-reads) will be made from these regions.

To minimize the cross-region write latency, [configure replication zones](/docs/v25.4/configure-replication-zones) such that the 3 US regions have voting replicas (that is, replicas that participate in [Raft quorum](/docs/v25.4/architecture/replication-layer#raft)). This will leave the 2 additional non-US regions with only [non-voting replicas](/docs/v25.4/architecture/replication-layer#non-voting-replicas).

For this example, you need to start a [`cockroach demo` cluster](/docs/v25.4/cockroach-demo) using the following command, which [defines the available cluster localities using the `--demo-locality` flag](/docs/v25.4/cockroach-demo#demo-locality):

```shell
cockroach demo --nodes=5 --demo-locality=region=us-east1:region=us-west1:region=us-central1:region=europe-west1:region=asia-northeast1
```

Next, enter the following statements at the SQL prompt. These statements will:

- Make the [`movr` database](/docs/v25.4/movr) into a [multi-region database](/docs/v25.4/multiregion-overview) by [setting the primary region](/docs/v25.4/alter-database#set-primary-region).
- Set the `movr` database to [survive region failures](/docs/v25.4/multiregion-survival-goals#survive-region-failures).
- Add the remaining regions defined at cluster start time to the multi-region `movr` database (for a total of 5).

```sql
ALTER DATABASE movr SET PRIMARY REGION "us-west1";
ALTER DATABASE movr ADD REGION "us-central1";
ALTER DATABASE movr ADD REGION "us-east1";
ALTER DATABASE movr ADD REGION "europe-west1";
ALTER DATABASE movr ADD REGION "asia-northeast1";
ALTER DATABASE movr SURVIVE REGION FAILURE;
```

By default, all tables in the `movr` database are [`REGIONAL` tables](/docs/v25.4/regional-tables):

```sql
SHOW TABLES;
```

```text
  schema_name |         table_name         | type  | owner | estimated_row_count |              locality
--------------+----------------------------+-------+-------+---------------------+--------------------------------------
  public      | promo_codes                | table | demo  |                1000 | REGIONAL BY TABLE IN PRIMARY REGION
  public      | rides                      | table | demo  |                 500 | REGIONAL BY TABLE IN PRIMARY REGION
  public      | user_promo_codes           | table | demo  |                   5 | REGIONAL BY TABLE IN PRIMARY REGION
  public      | users                      | table | demo  |                  50 | REGIONAL BY TABLE IN PRIMARY REGION
  public      | vehicle_location_histories | table | demo  |                1000 | REGIONAL BY TABLE IN PRIMARY REGION
  public      | vehicles                   | table | demo  |                  15 | REGIONAL BY TABLE IN PRIMARY REGION
(6 rows)
```

Now that the multi-region cluster is configured, we will use the following [`ALTER LOCALITY`](#alter-locality) statements to [configure replication zones](/docs/v25.4/configure-replication-zones) such that:

- The 3 US regions have voting replicas (that is, replicas that participate in [Raft quorum](/docs/v25.4/architecture/replication-layer#raft) ).
- The 2 additional regions (Europe and Japan) are left with only [non-voting replicas](/docs/v25.4/architecture/replication-layer#non-voting-replicas) that can be used to serve [follower reads](/docs/v25.4/follower-reads).

```sql

```

#### Failover regions

In this example we will use Zone Config Extensions to configure a multi-region [MovR](/docs/v25.4/movr) database so that if the [primary region](#set-primary-region) fails, enough [replicas](/docs/v25.4/architecture/distribution-layer) will be found in another region (the "failover" region) to take over for the primary region. We will set the [locality scope for this configuration to be `REGIONAL IN`](#parameters), which covers all [`REGIONAL BY TABLE IN {region}` tables](/docs/v25.4/alter-table#set-the-table-locality-to-regional-by-table) and all `{region}` [partitions](/docs/v25.4/partitioning) of [`REGIONAL BY ROW` tables](/docs/v25.4/alter-table#set-the-table-locality-to-regional-by-row).

<Note>
This functionality is already provided by the built-in [Secondary regions](/docs/v25.4/multiregion-overview#secondary-regions) feature. It is used here to show the flexibility of Zone Config Extensions. We strongly recommend using the [built-in multi-region features](/docs/v25.4/multiregion-overview) whenever possible.
</Note>

1. Set the database to have a `REGION` survival goal using [`ALTER DATABASE... SURVIVE REGION FAILURE`](/docs/v25.4/alter-database#survive-zone-region-failure):

    ```sql
     ALTER DATABASE movr SURVIVE REGION FAILURE;
    ```
2. Apply the `REGIONAL BY ROW` locality to the `movr.rides` table using the following statement:

    ```sql
     ALTER TABLE rides ADD COLUMN region crdb_internal_region AS (
      CASE WHEN city = 'amsterdam' THEN 'europe-west1'
           WHEN city = 'paris' THEN 'europe-west1'
           WHEN city = 'rome' THEN 'europe-west1'
           WHEN city = 'new york' THEN 'us-east1'
           WHEN city = 'boston' THEN 'us-east1'
           WHEN city = 'washington dc' THEN 'us-east1'
           WHEN city = 'san francisco' THEN 'us-west1'
           WHEN city = 'seattle' THEN 'us-west1'
           WHEN city = 'los angeles' THEN 'us-west1'
      END
     ) STORED;
     ALTER TABLE rides ALTER COLUMN region SET NOT NULL;
     ALTER TABLE rides SET LOCALITY REGIONAL BY ROW AS "region";
    ```
3. View the [zone configs](/docs/v25.4/configure-replication-zones) for the `movr.rides` table using [`SHOW ZONE CONFIGURATION`](/docs/v25.4/show-zone-configurations):

    ```sql
     SHOW ZONE CONFIGURATION FROM TABLE movr.rides;
    ```

    ```text
          target     |                                      raw_config_sql
     ----------------+-------------------------------------------------------------------------------------------
       DATABASE movr | ALTER DATABASE movr CONFIGURE ZONE USING
                     |     range_min_bytes = 134217728,
                     |     range_max_bytes = 536870912,
                     |     gc.ttlseconds = 90000,
                     |     num_replicas = 5,
                     |     num_voters = 5,
                     |     constraints = '{+region=europe-west1: 1, +region=us-east1: 1, +region=us-west1: 1}',
                     |     voter_constraints = '{+region=us-east1: 2}',
                     |     lease_preferences = '[[+region=us-east1]]'
     (1 row)
    ```

    Remember that [we configured `us-east1` to be our primary region during cluster setup](#use-zone-config-extensions). The output above confirms that `us-east1` is the primary region based on the values of the [`voter_constraints`](/docs/v25.4/configure-replication-zones#voter_constraints) and [`lease_preferences`](/docs/v25.4/configure-replication-zones#lease_preferences) keys.
4. Update the configuration to keep additional voting replicas and leaseholders in `us-west1`. We do this because we would like to configure `us-west1` to be the failover region for `us-east1`. The following SQL statement accomplishes this by configuring `us-east1` to keep additional voting replicas and leaseholders in `us-west1`. This means that if `us-east1` fails, it will fail over to `us-west1`.

    ```sql
     ALTER DATABASE movr ALTER LOCALITY REGIONAL IN "us-east1" CONFIGURE ZONE USING voter_constraints = '{+region=us-east1: 2, +region=us-west1: 2}', lease_preferences = '[[+region=us-east1], [+region=us-west1]]';
    ```
5. View the [zone configs](/docs/v25.4/configure-replication-zones) for the `movr.rides` table using [`SHOW ZONE CONFIGURATION`](/docs/v25.4/show-zone-configurations):

    ```sql
     SHOW ZONE CONFIGURATION FROM TABLE movr.rides;
    ```

    ```text
          target     |                                      raw_config_sql
     ----------------+-------------------------------------------------------------------------------------------
       DATABASE movr | ALTER DATABASE movr CONFIGURE ZONE USING
                     |     range_min_bytes = 134217728,
                     |     range_max_bytes = 536870912,
                     |     gc.ttlseconds = 90000,
                     |     num_replicas = 5,
                     |     num_voters = 5,
                     |     constraints = '{+region=europe-west1: 1, +region=us-east1: 1, +region=us-west1: 1}',
                     |     voter_constraints = '{+region=us-east1: 2, +region=us-west1: 2}',
                     |     lease_preferences = '[[+region=us-east1], [+region=us-west1]]'
     (1 row)
    ```

    The following changes are shown: - There are now 2 voting replicas stored in `us-west1`. - There is now a preference that if leases cannot be placed in `us-east1`, they should be placed in `us-west1`.

    Both of these changes combine to ensure that if `us-east1` goes down, the cluster will still be able to operate until some mitigation is in place.

To remove the zone config changes made in this example, [reset the Zone Config Extensions](#reset-a-regions-zone-config-extensions).

#### Reset a region's Zone Config Extensions

To reset the Zone Config Extension configuration applied to a region to the default settings, enter the following statement:

```sql
ALTER DATABASE movr ALTER LOCALITY CONFIGURE ZONE USING DEFAULT;
```

This will not reset any configuration created by the [multi-region abstractions](/docs/v25.4/multiregion-overview).
</Note>

To discard the Zone Config Extension settings from a region, enter the following statement:


<Note>
When you discard a zone configuration, the objects it was applied to will then inherit a configuration from an object "the next level up"; e.g., if the object whose configuration is being discarded is a table, it will use its parent database's configuration.

However, this statement will not remove any configuration created by the [multi-region abstractions](/docs/v25.4/multiregion-overview).
</Note>

        ```sql
        SET CLUSTER SETTING sql.defaults.multiregion_placement_policy.enabled = on;
        ```

<Note>
Use [`ALTER ROLE ALL SET {sessionvar} = {val}`](/docs/v25.4/alter-role#set-default-session-variable-values-for-all-users) instead of the `sql.defaults.*` [cluster settings](/docs/v25.4/cluster-settings). This allows you to set a default value for all users for any [session variable](/docs/v25.4/set-vars) that applies during login, making the `sql.defaults.*` cluster settings redundant.
</Note>

#### Create a database with the replica placement policy set to restricted

If you know at database creation time that you'd like to set the database's replica placement policy to ["restricted"](#parameters-restricted), you can do so in a [`CREATE DATABASE`](/docs/v25.4/create-database) statement as shown below:

```sql
CREATE DATABASE movr2 PRIMARY REGION "us-east1" REGIONS "us-west1", "europe-west1" PLACEMENT RESTRICTED;
```

```text
CREATE DATABASE
```

#### Set the replica placement policy to `RESTRICTED`

When you set the database's placement policy to [`RESTRICTED`](#parameters-restricted), you are saying that you want the underlying data to be restricted to the table or partition's [home region](/docs/v25.4/multiregion-overview#table-localities).

```sql
ALTER DATABASE movr PLACEMENT RESTRICTED;
```

```text
ALTER DATABASE PLACEMENT
```

#### Set the replica placement policy to `DEFAULT`

If previously you set the replica placement policy to [`RESTRICTED`](#set-the-replica-placement-policy-to-restricted), you can set it back to [the default](#parameters-default) by issuing the following statement:

```sql
ALTER DATABASE movr PLACEMENT DEFAULT;
```

```text
ALTER DATABASE PLACEMENT
```

### Configure survival goals

#### Survive zone failures

To change the survival goal of a multi-region database to survive zone failures, use the following statement:

```sql
ALTER DATABASE {db} SURVIVE ZONE FAILURE;
```

```text
ALTER DATABASE SURVIVE
```

<Note>
Surviving zone failures is the default setting for multi-region databases.
</Note>

For more information about the zone survival goal, see [Surviving zone failures](/docs/v25.4/multiregion-survival-goals#survive-zone-failures).

#### Survive region failures

To change the survival goal of a multi-region database to survive region failures, use the following statement:

```sql
ALTER DATABASE {db} SURVIVE REGION FAILURE;
```

```text
ALTER DATABASE SURVIVE
```

If you try to change a database with less than 3 [database regions](/docs/v25.4/multiregion-overview#database-regions) to survive region failures, the following error will be signalled:

```text
ERROR: at least 3 regions are required for surviving a region failure
SQLSTATE: 42602
HINT: you must add additional regions to the database using ALTER DATABASE mr ADD REGION <region_name>
```

For more information about the region survival goal, see [Surviving region failures](/docs/v25.4/multiregion-survival-goals#survive-region-failures).
