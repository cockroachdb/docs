The simple `SELECT` clause is the main SQL syntax to read and process existing data.

When used as a stand-alone statement, the simple `SELECT` clause is also called "the `SELECT` statement". However, it is also a [selection clause](/docs/v24.1/selection-queries#selection-clauses) that can be combined with other constructs to form more complex [selection queries](/docs/v24.1/selection-queries).

## Synopsis

<SelectClause4 />

<Tip>
The simple `SELECT` clause also has other applications not covered here, such as executing [functions](/docs/v24.1/functions-and-operators) like `SELECT current_timestamp();`.
</Tip>

## Required privileges

The user must have the `SELECT` [privilege](/docs/v24.1/security-reference/authorization#managing-privileges) on the tables used as operands.

## Eliminate duplicate rows

The `DISTINCT` subclause specifies to remove duplicate rows.

By default, or when `ALL` is specified, `SELECT` returns all the rows selected, without removing duplicates. When `DISTINCT` is specified, duplicate rows are eliminated.

Without `ON`, two rows are considered duplicates if they are equal on all the results computed by `SELECT`.

With `ON`, two rows are considered duplicates if they are equal only using the [scalar expressions](/docs/v24.1/scalar-expressions) listed with `ON`. When two rows are considered duplicates according to `DISTINCT ON`, the values from the first `FROM` row in the order specified by [`ORDER BY`](/docs/v24.1/order-by) are used to compute the remaining target expressions. If `ORDER BY` is not specified, CockroachDB will pick any one of the duplicate rows as first row, non-deterministically.


### Aggregate functions

[Aggregate functions](/docs/v24.1/functions-and-operators#aggregate-functions) perform calculations on retrieved rows.

#### Perform aggregate function on entire column

By using an aggregate function as a `target_elem`, you can perform the calculation on the entire column.

```sql
> SELECT MIN(revenue) FROM rides;
```

```text
  min
+------+
  0.00
(1 row)
```

You can also use the retrieved value as part of an expression. For example, you can use the result in the `WHERE` clause to select additional rows that were not part of the aggregate function itself:

```sql
> SELECT id, city, vehicle_id, rider_id
FROM rides
WHERE revenue = (
      SELECT
      MIN(revenue)
      FROM rides
);
```


#### Perform aggregate function on retrieved rows

By filtering the statement, you can perform the calculation only on retrieved rows:

```sql
> SELECT SUM(revenue) FROM rides WHERE city IN ('new york', 'chicago');
```

```text
    sum
+---------+
  4079.00
(1 row)
```

#### Filter columns fed into aggregate functions

You can use `FILTER (WHERE <Boolean expression>)` in the `target_elem` to filter which rows are processed by an aggregate function; those that return `FALSE` or `NULL` for the `FILTER` clause's Boolean expression are not fed into the aggregate function:

```sql
> SELECT count(*) AS unfiltered, count(*) FILTER (WHERE revenue > 50) AS filtered FROM rides;
```

```text
  unfiltered | filtered
+------------+----------+
         500 |      252
(1 row)
```

#### Create aggregate groups

Instead of performing aggregate functions on an the entire set of retrieved rows, you can split the rows into groups and then perform the aggregate function on each of them.

When creating aggregate groups, each column selected as a `target_elem` must be included in a `GROUP BY` clause.

For example:

```sql
> SELECT city, SUM(revenue) AS city_revenue FROM rides
WHERE city IN ('new york', 'chicago', 'seattle') GROUP BY city;
```

```text
    city   | city_revenue
+----------+--------------+
  chicago  |      1990.00
  new york |      2089.00
  seattle  |      2029.00
(3 rows)
```

<Note>
If the group is created on a primary key column, any column in the table can be selected as a `target_elem`. If a selected column is in a [subquery](/docs/v24.1/subqueries) that references a higher scope, a `GROUP BY` clause is not needed.
</Note>

#### Filter aggregate groups

To filter aggregate groups, use `HAVING`, which is the equivalent of the `WHERE` clause for aggregate groups, which must evaluate to a Boolean value.

For example:

```sql
> SELECT city, AVG(revenue) as avg FROM rides GROUP BY city
HAVING AVG(revenue) BETWEEN 50 AND 60;
```

```text
      city      |          avg
+---------------+-----------------------+
  amsterdam     |                 52.50
  boston        | 52.666666666666666667
  los angeles   | 55.951219512195121951
  minneapolis   | 55.146341463414634146
  washington dc | 58.756097560975609756
(5 rows)
```

#### Use aggregate functions in having clause

Aggregate functions can also be used in the `HAVING` clause without needing to be included as a `target_elem`.

For example:

```sql
> SELECT vehicle_id, city FROM rides WHERE city IN ('new york', 'chicago', 'seattle')
GROUP BY vehicle_id, city HAVING COUNT(vehicle_id) > 20;
```

```text
               vehicle_id              |   city
+--------------------------------------+----------+
  88888888-8888-4800-8000-000000000008 | chicago
  11111111-1111-4100-8000-000000000001 | new york

(3 rows)
```

#### Order aggregate function input rows by column

Non-commutative aggregate functions are sensitive to the order in which the rows are processed in the surrounding `SELECT` clause. To specify the order in which input rows are processed, you can add an [`ORDER BY`](/docs/v24.1/order-by) clause within the function argument list.

For example, suppose you want to create an array of `name` values, ordered alphabetically, and grouped by `city`. You can use the following statement to do so:

```sql
> SELECT city, array_agg(name ORDER BY name) AS users FROM users WHERE city IN ('new york', 'chicago', 'seattle') GROUP BY city;
```

### Select from a specific index

By using the explicit index annotation, you can override [CockroachDB's index selection](https://www.cockroachlabs.com/blog/index-selection-cockroachdb-2) and use a specific [index](/docs/v24.1/indexes) when reading from a named table.

<Note>
Index selection can impact [performance](/docs/v24.1/performance-best-practices-overview), but does not change the result of a query.
</Note>

##### Force index scan

To force a scan of a specific index:

```sql
SELECT * FROM table@my_idx;
```

This is equivalent to the longer expression:

```sql
SELECT * FROM table@{FORCE_INDEX=my_idx};
```

##### Force reverse scan

To force a reverse scan of a specific index:

```sql
SELECT * FROM table@{FORCE_INDEX=my_idx,DESC};
```

Forcing a reverse scan can help with [performance tuning](/docs/v24.1/performance-best-practices-overview). To choose an index and its scan direction:

```sql
SELECT * FROM table@{FORCE_INDEX=idx[,DIRECTION]};
```

where the optional `DIRECTION` is either `ASC` (ascending) or `DESC` (descending).

When a direction is specified, that scan direction is forced; otherwise the [cost-based optimizer](/docs/v24.1/cost-based-optimizer) is free to choose the direction it calculates will result in the best performance.

You can verify that the optimizer is choosing your desired scan direction using [`EXPLAIN (OPT)`](/docs/v24.1/explain#opt-option). For example, given the table

```sql
CREATE TABLE kv (K INT PRIMARY KEY, v INT);
```

you can check the scan direction with:

```sql
EXPLAIN (opt) SELECT * FROM users@{FORCE_INDEX=primary,DESC};
```

```text
                 text
+-------------------------------------+
  scan users,rev
   └── flags: force-index=primary,rev
(2 rows)
```

#### Force inverted index scan

To force a scan of any [inverted index](/docs/v24.1/inverted-indexes) of the hinted table:

```sql
SELECT * FROM table@{FORCE_INVERTED_INDEX};
```

The `FORCE_INVERTED_INDEX` hint does not allow specifying an inverted index. If no query plan can be generated, the query will result in the error:

```text
ERROR: could not produce a query plan conforming to the FORCE_INVERTED_INDEX hint
```

##### Force partial index scan

To force a [partial index scan](/docs/v24.1/partial-indexes), your statement must have a `WHERE` clause that implies the partial index filter.

```sql
CREATE TABLE t (
  a INT,
  INDEX idx (a) WHERE a > 0);
INSERT INTO t(a) VALUES (5);
SELECT * FROM t@idx WHERE a > 0;
```

```text
CREATE TABLE

Time: 13ms total (execution 12ms / network 0ms)

INSERT 1

Time: 22ms total (execution 21ms / network 0ms)

  a
-----
  5
(1 row)

Time: 1ms total (execution 1ms / network 0ms)
```

##### Force partial GIN index scan

To force a [partial GIN index](/docs/v24.1/inverted-indexes#partial-gin-indexes) scan, your statement must have a `WHERE` clause that:

- Implies the partial index.
- Constrains the GIN index scan.

```sql
DROP TABLE t;
CREATE TABLE t (
  j JSON,
  INVERTED INDEX idx (j) WHERE j->'a' = '1');
INSERT INTO t(j)
  VALUES ('{"a": 1}'),
         ('{"a": 3, "b": 2}'),
         ('{"a": 1, "b": 2}');
SELECT * FROM t@idx WHERE j->'a' = '1' AND j->'b' = '2';
```

```text
DROP TABLE

Time: 68ms total (execution 22ms / network 45ms)

CREATE TABLE

Time: 10ms total (execution 10ms / network 0ms)

INSERT 3

Time: 22ms total (execution 22ms / network 0ms)

         j
--------------------
  {"a": 1, "b": 2}
(1 row)

Time: 1ms total (execution 1ms / network 0ms)
```

##### Prevent full scan

- To prevent the optimizer from planning a full scan for a specific table, specify the `NO_FULL_SCAN` index hint. For example:

    ```sql
    SELECT * FROM table_name@{NO_FULL_SCAN};
    ```
- To prevent a full scan of a [partial index](/docs/v24.1/partial-indexes) for a specific table, you must specify `NO_FULL_SCAN` in combination with the index name using [`FORCE_INDEX`](/docs/v24.1/table-expressions#force-index-selection). For example:

    ```sql
    SELECT * FROM table_name@{FORCE_INDEX=index_name,NO_FULL_SCAN} WHERE b > 0;
    ```

    This forces a constrained scan of the partial index. If a constrained scan of the partial index is not possible, an error is returned.

<Tip>
For other ways to prevent full scans, refer to [Prevent the optimizer from planning full scans](/docs/v24.1/performance-best-practices-overview#prevent-the-optimizer-from-planning-full-scans).
</Tip>

## Advanced uses of `SELECT` clauses

CockroachDB supports numerous ways to combine results from `SELECT` clauses together.

See [Selection Queries](/docs/v24.1/selection-queries) for details. A few examples follow.

### Sorting and limiting query results

To order the results of a `SELECT` clause or limit the number of rows in the result, you can combine it with `ORDER BY` or `LIMIT` / `OFFSET` to form a [selection query](/docs/v24.1/selection-queries) or [subquery](/docs/v24.1/table-expressions#use-a-subquery).

See [Ordering Query Results](/docs/v24.1/order-by) and [Limiting Query Results](/docs/v24.1/limit-offset) for more details.

<Note>
When `ORDER BY` is not included in a query, rows are not sorted by any consistent criteria. Instead, CockroachDB returns them as the coordinating node receives them. Also, CockroachDB sorts [`NULL` values](/docs/v24.1/null-handling#nulls-and-sorting) first with `ASC` and last with `DESC`. This differs from PostgreSQL, which sorts `NULL` values last with `ASC` and first with `DESC`.
</Note>

### Combining results from multiple queries

Results from two or more queries can be combined together as follows:

- Using [`JOIN` expressions](/docs/v24.1/joins) to combine rows according to conditions on specific columns.
- Using [set operations](/docs/v24.1/selection-queries#set-operations) to combine rows using inclusion/exclusion rules.

### Row-level locking for concurrency control with `SELECT FOR UPDATE`

`SELECT... FOR UPDATE` exclusively locks the rows returned by a [selection query](/docs/v24.1/selection-queries), such that other transactions trying to access those rows must wait for the transaction that locked the rows to commit or rollback.

`SELECT... FOR UPDATE` can be used to:

- Strengthen the isolation of a [`READ COMMITTED`](/docs/v24.1/read-committed) transaction. If you need to read and later update a row within a transaction, use `SELECT... FOR UPDATE` to acquire an exclusive lock on the row. This guarantees data integrity between the transaction's read and write operations. For details, see [Locking reads](/docs/v24.1/read-committed#locking-reads).
- Order [`SERIALIZABLE`](/docs/v24.1/demo-serializable) transactions by controlling concurrent access to one or more rows of a table. These other transactions are placed into a queue based on when they tried to read the values of the locked rows.

    Because this queueing happens during the read operation, the [thrashing](https://wikipedia.org/wiki/Thrashing_(computer_science)) that would otherwise occur if multiple concurrently executing transactions attempt to `SELECT` the same data and then `UPDATE` the results of that selection is prevented. By preventing thrashing, `SELECT... FOR UPDATE` also prevents [transaction retries](/docs/v24.1/transactions#transaction-retries) that would otherwise occur due to [contention](/docs/v24.1/performance-best-practices-overview#transaction-contention).

    As a result, using `SELECT... FOR UPDATE` leads to increased throughput and decreased tail latency for contended operations.

Note that using `SELECT... FOR UPDATE` does not completely eliminate the chance of [serialization errors](/docs/v24.1/transaction-retry-error-reference). These errors can also arise due to [time uncertainty](/docs/v24.1/architecture/transaction-layer#transaction-conflicts). To eliminate the need for application-level retry logic, in addition to `SELECT FOR UPDATE` your application also needs to use a [driver that implements automatic retry handling](/docs/v24.1/transaction-retry-error-reference#client-side-retry-handling).

<Note>
By default, CockroachDB uses the `SELECT... FOR UPDATE` locking mechanism during the initial row scan performed in [`UPDATE`](/docs/v24.1/update) and [`UPSERT`](/docs/v24.1/upsert) statement execution. To turn off implicit `SELECT... FOR UPDATE` locking for `UPDATE` and `UPSERT` statements, set the `enable_implicit_select_for_update` [session variable](/docs/v24.1/set-vars) to `false`.
</Note>

For an example, see [`FOR UPDATE` and `FOR SHARE`](/docs/v24.1/select-for-update).
