Cluster settings apply to all nodes of a CockroachDB cluster and control, for example, whether or not to share diagnostic details with Cockroach Labs as well as advanced options for debugging and cluster tuning.

<Note>
In contrast to cluster-wide settings, node-level settings apply to a single node. They are defined by flags passed to the `cockroach start` command when starting a node and cannot be changed without stopping and restarting the node. For more details, see [Start a Node](/docs/v24.1/cockroach-start).
</Note>

This page provides information on:

- [Available settings](#settings)
- How to [view current cluster settings](#view-current-cluster-settings)
- How to [change a cluster setting](#change-a-cluster-setting)
- [Sensitive settings](#sensitive-settings)

## Settings

<Warning>
These cluster settings have a broad impact on CockroachDB internals and affect all applications, workloads, and users running on a CockroachDB cluster. For some settings, a [session setting](/docs/v24.1/set-vars#supported-variables) could be a more appropriate scope.
</Warning>

<Note>
Use [`ALTER ROLE ALL SET {sessionvar} = {val}`](/docs/v24.1/alter-role#set-default-session-variable-values-for-all-users) instead of the `sql.defaults.*` [cluster settings](/docs/v24.1/cluster-settings). This allows you to set a default value for all users for any [session variable](/docs/v24.1/set-vars) that applies during login, making the `sql.defaults.*` cluster settings redundant.
</Note>

## View current cluster settings

Use the [`SHOW CLUSTER SETTING`](/docs/v24.1/show-cluster-setting) statement.

## Change a cluster setting

Cluster settings can be updated via SQL command while the cluster is running. Use the [`SET CLUSTER SETTING`](/docs/v24.1/set-cluster-setting) statement.

Before changing a cluster setting, note the following:

- Changing a cluster setting is not instantaneous, as the change must be propagated to other nodes in the cluster.
- Do not change cluster settings while [upgrading to a new version of CockroachDB](/docs/v24.1/upgrade-cockroach-version). Wait until all nodes have been upgraded before you make the change.

## Sensitive settings

You can prevent users without sufficient permissions from viewing the values of cluster settings that CockroachDB classifies as sensitive.

By default, users with the `VIEWCLUSTERSETTING` privilege can view the values of all settings displayed when using the [`SHOW CLUSTER SETTING`](/docs/v24.1/show-cluster-setting) statement.

If you enable the option to redact sensitive settings, the sensitive setting values are hidden from those users, and visible only to users with the `admin` role or the `MODIFYCLUSTERSETTING` privilege.

To enable this redaction of sensitive setting values, set the cluster setting [`server.redact_sensitive_settings.enabled`](/docs/v24.1/cluster-settings#setting-server-redact-sensitive-settings-enabled) to `true`:

```sql
SET CLUSTER SETTING server.redact_sensitive_settings.enabled = 'true';
```

The table summarizes when sensitive setting values are visible or redacted:

<table><thead><tr><th>User attribute</th><th>Redaction disabled</th><th>Redaction enabled</th></tr></thead><tbody><tr><td><code>admin</code> role</td><td>visible</td><td>visible</td></tr><tr><td><code>MODIFYCLUSTERSETTING</code><a href="/docs/v24.1/security-reference/authorization#privileges">system-level privilege</a></td><td>visible</td><td>visible</td></tr><tr><td><code>VIEWCLUSTERSETTING</code><a href="/docs/v24.1/security-reference/authorization#privileges">system-level privilege</a></td><td>visible</td><td><strong>redacted</strong></td></tr><tr><td>None of the above attributes</td><td>not visible</td><td>not visible</td></tr></tbody></table>

The following are sensitive settings whose values are redacted:

- [`server.oidc_authentication.client_id`](/docs/v24.1/cluster-settings#setting-server-oidc-authentication-client-id)
- [`server.oidc_authentication.client_secret`](/docs/v24.1/cluster-settings#setting-server-oidc-authentication-client-secret)
