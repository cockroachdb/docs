The `CANCEL JOB` [statement](/docs/v24.1/sql-statements) lets you stop long-running jobs, which include:

- [`IMPORT`](/docs/v24.1/import-into) jobs
- [`BACKUP`](/docs/v24.1/backup) and [`RESTORE`](/docs/v24.1/restore) jobs
- [User-created table statistics](/docs/v24.1/create-statistics) jobs
- [Automatic table statistics](/docs/v24.1/cost-based-optimizer#table-statistics) jobs
- [Changefeeds](/docs/v24.1/create-changefeed)
- [Scheduled backup](/docs/v24.1/manage-a-backup-schedule) jobs
- [Schema change](/docs/v24.1/online-schema-changes) jobs (see [Known limitations](#known-limitations) for exceptions)

## Known limitations

- To avoid transaction states that cannot properly [roll back](/docs/v24.1/rollback-transaction), the following statements cannot be cancelled with [`CANCEL JOB`](/docs/v24.1/cancel-job):
    - `DROP` statements (e.g., [`DROP TABLE`](/docs/v24.1/drop-table) ).
    - `ALTER... RENAME` statements (e.g., [`ALTER TABLE... RENAME TO`](/docs/v24.1/alter-table#rename-to) ).
    - [`CREATE TABLE... AS`](/docs/v24.1/create-table-as) statements.
    - [`ALTER TYPE`](/docs/v24.1/alter-type) statements, except for those that drop values.
- When an Enterprise [`RESTORE`](/docs/v24.1/restore) is canceled, partially restored data is properly cleaned up. This can have a minor, temporary impact on cluster performance.

## Required privileges

To cancel a job, the user must be a member of the `admin` role or must have the [`CONTROLJOB`](/docs/v24.1/create-user#create-a-user-that-can-pause-resume-and-cancel-non-admin-jobs) [role option](/docs/v24.1/security-reference/authorization#role-options) set. Non-admin users cannot cancel admin users' jobs.

For changefeeds, users with the [`CHANGEFEED`](/docs/v24.1/create-changefeed#required-privileges) privilege on a set of tables can cancel changefeed jobs running on those tables.

## Examples

### Cancel a single job

```sql
> SHOW JOBS;
```

```text
+----------------+---------+------------------------------------------------+...
|       id       |  type   |               description                      |...
+----------------+---------+------------------------------------------------+...
| 27536791415282 | RESTORE | RESTORE db.* FROM 'azure-blob://backup/db/tbl' |...
+----------------+---------+------------------------------------------------+...
```

```sql
> CANCEL JOB 27536791415282;
```

### Cancel multiple jobs

To cancel multiple jobs, nest a [`SELECT` clause](/docs/v24.1/select-clause) that retrieves `job_id`(s) inside the `CANCEL JOBS` statement:

```sql
> CANCEL JOBS (WITH x AS (SHOW JOBS) SELECT job_id FROM x
      WHERE user_name = 'maxroach');
```

All jobs created by `maxroach` will be cancelled.

### Cancel by job type

To cancel all jobs by the type of job, use the `CANCEL ALL {job} JOBS` statement. You can cancel all `BACKUP`, `RESTORE`, `CHANGEFEED`, `IMPORT` jobs using this statement, for example:

```sql
CANCEL ALL BACKUP JOBS;
```

### Cancel automatic table statistics jobs

Canceling an automatic table statistics job is not useful since the system will automatically restart the job immediately. To permanently disable automatic table statistics jobs, disable the `sql.stats.automatic_collection.enabled` [cluster setting](/docs/v24.1/cluster-settings):

```sql
> SET CLUSTER SETTING sql.stats.automatic_collection.enabled = false;
```

### Cancel jobs for a schedule

To cancel jobs for a specific [backup schedule](/docs/v24.1/create-schedule-for-backup), use the schedule's `id`:

```sql
> CANCEL JOBS FOR SCHEDULE 590204387299262465;
```

```text
CANCEL JOBS FOR SCHEDULES 1
```

You can also CANCEL multiple schedules by nesting a [`SELECT` clause](/docs/v24.1/select-clause) that retrieves `id`(s) inside the `CANCEL JOBS` statement:

```sql
> CANCEL JOBS FOR SCHEDULES WITH x AS (SHOW SCHEDULES) SELECT id FROM x WHERE label = 'test_schedule';
```

```text
CANCEL JOBS FOR SCHEDULES 2
```
