CockroachDB provides built-in [**high availability (HA)**](#high-availability) features and [**disaster recovery (DR)**](#disaster-recovery) tooling to achieve operational resiliency in various deployment [topologies](/docs/v24.3/topology-patterns) and use cases.

- HA features ensure continuous access to data without interruption even in the presence of [failures](/docs/v24.3/demo-cockroachdb-resilience) or disruptions to maximize uptime.
- DR tools allow for recovery from major incidents to minimize downtime and data loss.

<Frame>
  ![Diagram showing how HA features and DR tools create a resilient CockroachDB deployment.](/images/v24.3/resilience.svg)
</Frame>

You can balance required SLAs and recovery objectives with the cost and management of each of these features to build a resilient deployment.

- **Recovery Point Objective (RPO)**: The maximum amount of data loss (measured by time) that an organization can tolerate.
- **Recovery Time Objective (RTO)**: The maximum length of time it should take to restore normal operations following an outage.

<Tip>
For a practical guide on how CockroachDB uses Raft to replicate, distribute, and rebalance data, refer to the [CockroachDB Resilience demo](/docs/v24.3/demo-cockroachdb-resilience).
</Tip>

## High availability

- [**Multi-active availability**](/docs/v24.3/multi-active-availability): CockroachDB's built-in [Raft replication](/docs/v24.3/architecture/replication-layer) stores data safely and consistently on multiple nodes to ensure no downtime even during a temporary node outage. [Replication controls](/docs/v24.3/configure-replication-zones) allow you to configure the number and location of [replicas](/docs/v24.3/architecture/glossary#replica) to suit a deployment.
    - For more detail on planning for single-region or multi-region recovery, refer to [Single-region survivability planning](/docs/v24.3/disaster-recovery-planning#single-region-survivability-planning) or [Multi-region survivability planning](/docs/v24.3/disaster-recovery-planning#multi-region-survivability-planning).
- [**Advanced fault tolerance**](/docs/v24.3/demo-cockroachdb-resilience): Capabilities built in to CockroachDB to perform routine maintenance operations with minimal impact to foreground performance. For example, [online schema changes](/docs/v24.3/online-schema-changes), [write-ahead log failover](/docs/v24.3/cockroach-start#write-ahead-log-wal-failover).
- [**Logical data replication (LDR)**](/docs/v24.3/logical-data-replication-overview) (Preview): A cross-cluster replication tool between active CockroachDB clusters, which supports a range of topologies. LDR provides eventually consistent, table-level replication between the clusters. Individually, each active cluster uses CockroachDB multi-active availability to achieve low, single-region write latency with transactionally consistent writes using Raft replication.

## Disaster recovery

- [**Backup and point-in-time restore**](/docs/v24.3/take-backups-with-revision-history-and-restore-from-a-point-in-time): Point-in-time backup and restore allows you to roll back to a specific point in time. Multiple supported [cloud storage](/docs/v24.3/use-cloud-storage) providers means that you can store backups in your chosen provider. [Incremental backups](/docs/v24.3/take-full-and-incremental-backups) allow you to configure backup frequency for lower [RPO](/docs/v24.3/disaster-recovery-overview).
- [**Physical cluster replication (PCR)**](/docs/v24.3/physical-cluster-replication-overview): A cross-cluster replication tool between an active primary CockroachDB cluster and a passive standby CockroachDB cluster. PCR provides transactionally consistent full-cluster replication.

### Choose a DR strategy

CockroachDB is designed to recover automatically; however, building backups or PCR into your DR plan protects against unforeseen incidents.
