The `BIT` and `VARBIT` [data types](/docs/v24.3/data-types) stores bit arrays. With `BIT`, the length is fixed; with `VARBIT`, the length can be variable.

## Aliases

The name `BIT VARYING` is an alias for `VARBIT`.

## Syntax

Bit array constants are expressed as literals. For example, `B'100101'` denotes an array of 6 bits.

For more information about bit array constants, see the [constants documentation on bit array literals](/docs/v24.3/sql-constants#bit-array-literals).

For usage, see the [Example](#example) below.

## Size

The number of bits in a `BIT` value is determined as follows:

<table><thead><tr><th>Type declaration</th><th>Logical size</th></tr></thead><tbody><tr><td>BIT</td><td>1 bit</td></tr><tr><td>BIT(N)</td><td>N bits</td></tr><tr><td>VARBIT</td><td>variable with no maximum</td></tr><tr><td>VARBIT(N)</td><td>variable with a maximum of N bits</td></tr></tbody></table>

The effective size of a `BIT` value is larger than its logical number of bits by a bounded constant factor. Internally, CockroachDB stores bit arrays in increments of 64 bits plus an extra integer value to encode the length.

The total size of a `BIT` value can be arbitrarily large, but it is recommended to keep values under 1 MB to ensure performance. Above that threshold, [write amplification](https://wikipedia.org/wiki/Write_amplification) and other considerations may cause significant performance degradation.

## Example

```sql
> CREATE TABLE b (x BIT, y BIT(3), z VARBIT, w VARBIT(3));
```

```sql
> SHOW COLUMNS FROM b;
```

```text
  column_name | data_type | is_nullable | column_default | generation_expression |  indices  | is_hidden
+-------------+-----------+-------------+----------------+-----------------------+-----------+-----------+
  x           | BIT       |    true     | NULL           |                       | {primary} |   false
  y           | BIT(3)    |    true     | NULL           |                       | {primary} |   false
  z           | VARBIT    |    true     | NULL           |                       | {primary} |   false
  w           | VARBIT(3) |    true     | NULL           |                       | {primary} |   false
  rowid       | INT       |    false    | unique_rowid() |                       | {primary} |   true
```

```sql
> INSERT INTO b(x, y, z, w) VALUES (B'1', B'101', B'1', B'1');
```

```sql
> SELECT * FROM b;
```

```text
  x |  y  | z | w
+---+-----+---+---+
  1 | 101 | 1 | 1
```

For type `BIT`, the value must match exactly the specified size:

```sql
> INSERT INTO b(x) VALUES (B'101');
```

```text
pq: bit string length 3 does not match type BIT
```

```sql
> INSERT INTO b(y) VALUES (B'10');
```

```text
pq: bit string length 2 does not match type BIT(3)
```

For type `VARBIT`, the value must not be larger than the specified maximum size:

```sql
> INSERT INTO b(w) VALUES (B'1010');
```

```text
pq: bit string length 4 too large for type VARBIT(3)
```

## Supported casting and conversion

`BIT` values can be [cast](/docs/v24.3/data-types#data-type-conversions-and-casts) to any of the following data types:

<table><thead><tr><th>Type</th><th>Details</th></tr></thead><tbody><tr><td><code>INT</code></td><td>Converts the bit array to the corresponding numeric value, interpreting the bits as if the value was encoded using <a href="https://wikipedia.org/wiki/Two%27s_complement">two's complement</a>. If the bit array is larger than the integer type, excess bits on the left are ignored. For example, <code>B'1010'::INT</code> equals <code>10</code>.</td></tr><tr><td><code>STRING</code></td><td>Prints out the binary digits as a string. This recovers the literal representation. For example, <code>B'1010'::STRING</code> equals <code>'1010'</code>.</td></tr></tbody></table>
