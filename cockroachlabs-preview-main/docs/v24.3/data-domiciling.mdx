As you scale your usage of [multi-region clusters](/docs/v24.3/multiregion-overview), you may need to keep certain subsets of data in specific localities. Keeping specific data on servers in specific geographic locations is also known as *data domiciling*.

CockroachDB has basic support for data domiciling in multi-region clusters using the [`ALTER DATABASE... ADD SUPER REGION`](/docs/v24.3/alter-database#add-super-region) statement.

<Warning>
Using CockroachDB as part of your approach to data domiciling has several limitations. For more information, see [Known limitations](#known-limitations).
</Warning>

## Overview

This page has instructions for data domiciling in [multi-region clusters](/docs/v24.3/multiregion-overview). At a high level, this process involves:

1. Controlling the placement of specific row or table data using regional tables with the [`REGIONAL BY ROW`](/docs/v24.3/table-localities#regional-by-row-tables) and [`REGIONAL BY TABLE`](/docs/v24.3/table-localities#regional-tables) clauses.
2. Further restricting where the data in those regional tables is stored using the [`ALTER DATABASE... ADD SUPER REGION`](/docs/v24.3/alter-database#add-super-region), which creates a set of [database regions](/docs/v24.3/multiregion-overview#database-regions) such that [regional tables](/docs/v24.3/table-localities#regional-tables) and [regional by row tables](/docs/v24.3/table-localities#regional-by-row-tables) whose [home regions](/docs/v24.3/table-localities) are in the super region will have all of their [replicas](/docs/v24.3/architecture/overview#architecture-replica) stored *only* in regions that are members of the super region. For more information, see [Super regions](/docs/v24.3/multiregion-overview#super-regions).

<Note>
An alternative method to the [`ALTER DATABASE... ADD SUPER REGION`](/docs/v24.3/alter-database#add-super-region) statement is to use the [`ALTER DATABASE... PLACEMENT RESTRICTED`](/docs/v24.3/alter-database#placement) statement.

**`PLACEMENT RESTRICTED` is not recommended, and is documented for backwards compatibility. Most users should use `ADD SUPER REGION`, which allows for region survival as well as providing data placement.**
</Note>

## Before you begin

This page assumes you are already familiar with:

- CockroachDB's [multi-region SQL abstractions](/docs/v24.3/multiregion-overview). If you are not using them, the instructions on this page will not apply.
- The fact that CockroachDB stores your data in [a distributed key-value store, which is split into chunks called ranges](/docs/v24.3/architecture/distribution-layer#overview).

## Example

In the following example, you will go through the process of configuring the [MovR](/docs/v24.3/movr) data set using [multi-region SQL statements](/docs/v24.3/multiregion-overview).

Then, as part of implementing a data domiciling strategy, you will apply restricted replica settings in [Step 4](#step-4-apply-stricter-replica-placement-settings) using the [`ALTER DATABASE... ADD SUPER REGION`](/docs/v24.3/alter-database#add-super-region): It works with databases with [zone survival goals](/docs/v24.3/multiregion-survival-goals#survive-zone-failures) or [region survival goals](/docs/v24.3/multiregion-survival-goals#survive-region-failures). If you need region survival goals, you must use [super regions](/docs/v24.3/multiregion-overview#super-regions).

Finally, you will verify that the resulting replica placements are as expected using the [critical nodes status endpoint](/docs/v24.3/monitoring-and-alerting#critical-nodes-endpoint).

For the purposes of this example, the data domiciling requirement is to configure a multi-region deployment of the [MovR database](/docs/v24.3/movr) such that data for EU-based users, vehicles, etc. is being stored on CockroachDB nodes running in EU localities.

### Step 2. Apply multi-region SQL abstractions

Execute the following statements to set the [database regions](/docs/v24.3/multiregion-overview#database-regions). This information is necessary so that CockroachDB can later move data around to optimize access to particular data from particular regions.

```sql
ALTER DATABASE movr PRIMARY REGION "europe-west1";
ALTER DATABASE movr ADD REGION "us-east1";
ALTER DATABASE movr ADD REGION "us-west1";
```

Because the data in `promo_codes` is not updated frequently (a.k.a., "read-mostly"), and needs to be available from any region, the right table locality is [`GLOBAL`](/docs/v24.3/table-localities#global-tables).

```sql
ALTER TABLE promo_codes SET locality GLOBAL;
```

Next, alter the `user_promo_codes` table to have a foreign key into the global `promo_codes` table. This will enable fast reads of the `promo_codes.code` column from any region in the cluster.

```sql
ALTER TABLE user_promo_codes
  ADD CONSTRAINT user_promo_codes_code_fk
    FOREIGN KEY (code)
    REFERENCES promo_codes (code)
    ON UPDATE CASCADE;
```

All of the tables except `promo_codes` contain rows which are partitioned by region, and updated very frequently. For these tables, the right table locality for optimizing access to their data is [`REGIONAL BY ROW`](/docs/v24.3/table-localities#regional-by-row-tables).

Apply this table locality to the remaining tables. These statements use a `CASE` statement to put data for a given city in the right region and can take around 1 minute to complete for each table.

- `rides`

    ```sql
    ALTER TABLE rides ADD COLUMN region crdb_internal_region AS (
      CASE WHEN city = 'amsterdam' THEN 'europe-west1'
           WHEN city = 'paris' THEN 'europe-west1'
           WHEN city = 'rome' THEN 'europe-west1'
           WHEN city = 'new york' THEN 'us-east1'
           WHEN city = 'boston' THEN 'us-east1'
           WHEN city = 'washington dc' THEN 'us-east1'
           WHEN city = 'san francisco' THEN 'us-west1'
           WHEN city = 'seattle' THEN 'us-west1'
           WHEN city = 'los angeles' THEN 'us-west1'
      END
    ) STORED;
    ALTER TABLE rides ALTER COLUMN REGION SET NOT NULL;
    ALTER TABLE rides SET LOCALITY REGIONAL BY ROW AS "region";
    ```
- `user_promo_codes`

    ```sql
    ALTER TABLE user_promo_codes ADD COLUMN region crdb_internal_region AS (
      CASE WHEN city = 'amsterdam' THEN 'europe-west1'
           WHEN city = 'paris' THEN 'europe-west1'
           WHEN city = 'rome' THEN 'europe-west1'
           WHEN city = 'new york' THEN 'us-east1'
           WHEN city = 'boston' THEN 'us-east1'
           WHEN city = 'washington dc' THEN 'us-east1'
           WHEN city = 'san francisco' THEN 'us-west1'
           WHEN city = 'seattle' THEN 'us-west1'
           WHEN city = 'los angeles' THEN 'us-west1'
      END
    ) STORED;
    ALTER TABLE user_promo_codes ALTER COLUMN REGION SET NOT NULL;
    ALTER TABLE user_promo_codes SET LOCALITY REGIONAL BY ROW AS "region";
    ```
- `users`

    ```sql

      CASE WHEN city = 'amsterdam' THEN 'europe-west1'
           WHEN city = 'paris' THEN 'europe-west1'
           WHEN city = 'rome' THEN 'europe-west1'
           WHEN city = 'new york' THEN 'us-east1'
           WHEN city = 'boston' THEN 'us-east1'
           WHEN city = 'washington dc' THEN 'us-east1'
           WHEN city = 'san francisco' THEN 'us-west1'
           WHEN city = 'seattle' THEN 'us-west1'
           WHEN city = 'los angeles' THEN 'us-west1'
      END
    ) STORED;
    ALTER TABLE users ALTER COLUMN REGION SET NOT NULL;

    ```
- `vehicle_location_histories`

    ```sql
    ALTER TABLE vehicle_location_histories ADD COLUMN region crdb_internal_region AS (
      CASE WHEN city = 'amsterdam' THEN 'europe-west1'
           WHEN city = 'paris' THEN 'europe-west1'
           WHEN city = 'rome' THEN 'europe-west1'
           WHEN city = 'new york' THEN 'us-east1'
           WHEN city = 'boston' THEN 'us-east1'

    ALTER TABLE vehicle_location_histories ALTER COLUMN REGION SET NOT NULL;
    ```

- `vehicles`

    ```sql

           WHEN city = 'paris' THEN 'europe-west1'
    ```

### Step 3. View noncompliant replicas

Next, check the [critical nodes status endpoint](/docs/v24.3/monitoring-and-alerting#critical-nodes-endpoint) to see which ranges are still not in compliance with your desired domiciling: that data on EU-based entities (users, etc.) does not leave EU-based nodes.

On a small demo cluster like this one, the data movement from the previous step should finish quickly; on larger clusters, the rebalancing process may take longer. For more information about the performance considerations of rebalancing data in multi-region clusters, see [Performance considerations](/docs/v24.3/migrate-to-multiregion-sql#performance-considerations).

With the default settings, you should expect some replicas in the cluster to be violating this constraint. Those replicas will appear in the `violatingConstraints` field of the output.

This is because [non-voting replicas](/docs/v24.3/architecture/replication-layer#non-voting-replicas) are enabled by default in [multi-region clusters](/docs/v24.3/multiregion-overview) to enable stale reads of data in [regional tables](/docs/v24.3/regional-tables) from outside those tables' [home regions](/docs/v24.3/multiregion-overview#table-localities). For many use cases, this is preferred, but it keeps you from meeting the domiciling requirements for this example.

### Step 4. Apply stricter replica placement settings

<Tabs>
<Tab title="(Recommended) Use ADD SUPER REGION">

To ensure that data on EU-based users, vehicles, etc. from [`REGIONAL BY ROW` tables](/docs/v24.3/regional-tables#regional-by-row-tables) is stored only on EU-based nodes in the cluster, you can use [Super regions](/docs/v24.3/multiregion-overview#super-regions). This will ensure that [regional tables](/docs/v24.3/table-localities#regional-tables) and [regional by row tables](/docs/v24.3/table-localities#regional-by-row-tables) whose [home regions](/docs/v24.3/table-localities) are in the super region will have all of their [replicas](/docs/v24.3/architecture/overview#architecture-replica) stored *only* in regions that are members of the super region.

To use this statement, you must set the `enable_super_regions` [session setting](/docs/v24.3/set-vars):

```sql
ALTER ROLE ALL SET enable_super_regions = on;
```

Next, use the [`ALTER DATABASE... ADD SUPER REGION`](/docs/v24.3/alter-database#add-super-region) statement:

```sql
ALTER DATABASE movr ADD SUPER REGION "europe" VALUES "europe-west1";
```

You have now created a super region with only one region. The updated replica placement settings should start to apply immediately.

<Note>
[`ALTER DATABASE... ADD SUPER REGION`](/docs/v24.3/alter-database#add-super-region) does not affect the replica placement for [global tables](/docs/v24.3/global-tables), which are designed to provide fast, up-to-date reads from all [database regions](/docs/v24.3/multiregion-overview#database-regions).
</Note>

</Tab>
<Tab title="(Not recommended) Use PLACEMENT RESTRICTED">

**This method is not recommended, and is documented for backwards compatibility. Most users should use `ALTER DATABASE... ADD SUPER REGION` which allows for region survival as well as providing data placement.**

To ensure that data on EU-based users, vehicles, etc. from [`REGIONAL BY ROW` tables](/docs/v24.3/regional-tables#regional-by-row-tables) is stored only on EU-based nodes in the cluster, you must disable the use of [non-voting replicas](/docs/v24.3/architecture/replication-layer#non-voting-replicas) on all of the [regional tables](/docs/v24.3/regional-tables) in this database. You can do this using the [`ALTER DATABASE... PLACEMENT RESTRICTED`](/docs/v24.3/alter-database#placement) statement.

To use this statement, you must set the `enable_multiregion_placement_policy` [session setting](/docs/v24.3/set-vars) or the `sql.defaults.multiregion_placement_policy.enabled` [cluster setting](/docs/v24.3/cluster-settings):

```sql
SET enable_multiregion_placement_policy=on;
```

Next, use the [`ALTER DATABASE... PLACEMENT RESTRICTED`](/docs/v24.3/alter-database#placement) statement to disable non-voting replicas for regional tables:

```sql
ALTER DATABASE movr PLACEMENT RESTRICTED;
```

The restricted replica placement settings should start to apply immediately.

If you want to do data domiciling for databases with [region survival goals](/docs/v24.3/multiregion-survival-goals#survive-region-failures) using the higher-level [multi-region abstractions](/docs/v24.3/multiregion-overview), you must use [super regions](/docs/v24.3/multiregion-overview#super-regions). Using [`ALTER DATABASE... PLACEMENT RESTRICTED`](/docs/v24.3/alter-database#placement) will not work for databases that are set up with region survival goals.

<Note>
[`ALTER DATABASE... PLACEMENT RESTRICTED`](/docs/v24.3/alter-database#placement) does not affect the replica placement for [global tables](/docs/v24.3/global-tables), which are designed to provide fast, up-to-date reads from all [database regions](/docs/v24.3/multiregion-overview#database-regions).
</Note>

<Note>
Use [`ALTER ROLE ALL SET {sessionvar} = {val}`](/docs/v24.3/alter-role#set-default-session-variable-values-for-all-users) instead of the `sql.defaults.*` [cluster settings](/docs/v24.3/cluster-settings). This allows you to set a default value for all users for any [session variable](/docs/v24.3/set-vars) that applies during login, making the `sql.defaults.*` cluster settings redundant.
</Note>

</Tab>
</Tabs>

### Step 5. Verify updated replica placement

Now that you have restricted the placement of non-voting replicas for all [regional tables](/docs/v24.3/regional-tables), you can check the [critical nodes status endpoint](/docs/v24.3/monitoring-and-alerting#critical-nodes-endpoint) to see the effects. In a few seconds, you should see that the `violatingConstraints` key in the JSON response shows that there are no longer any replicas violating their constraints:

```shell
curl -X POST http://localhost:8080/_status/critical_nodes
```

```json
{
  "criticalNodes": [
  ],
  "report": {
    "underReplicated": [
    ],
    "overReplicated": [
    ],
    "violatingConstraints": [
    ],
    "unavailable": [
    ],
    "unavailableNodeIds": [
    ]
  }
}
```

The output above shows that there are no replicas that do not meet the data domiciling goal. As described above, [`ALTER DATABASE... PLACEMENT RESTRICTED`](/docs/v24.3/alter-database#placement) does not affect the replica placement for [`GLOBAL` tables](/docs/v24.3/global-tables), so these replicas are considered to be in compliance.

Now that you have verified that the system is configured to meet the domiciling requirement, it's a good idea to check the [critical nodes status endpoint](/docs/v24.3/monitoring-and-alerting#critical-nodes-endpoint) on a regular basis (via automation of some kind) to ensure that the requirement continues to be met.

<Note>
The steps above are necessary but not sufficient to accomplish a data domiciling solution using CockroachDB. Be sure to review the [limitations of CockroachDB for data domiciling](#known-limitations) and design your total solution with those limitations in mind.
</Note>

## Known limitations

Using CockroachDB as part of your approach to data domiciling has several limitations:

- When columns are [indexed](/docs/v24.3/indexes), a subset of data from the indexed columns may appear in [meta ranges](/docs/v24.3/architecture/distribution-layer#meta-ranges) or other system tables. CockroachDB synchronizes these system ranges and system tables across nodes. This synchronization does not respect any multi-region settings applied via either the [multi-region SQL statements](/docs/v24.3/multiregion-overview), or the low-level [zone configs](/docs/v24.3/configure-replication-zones) mechanism.
- [Zone configs](/docs/v24.3/configure-replication-zones) can be used for data placement but these features were historically built for performance, not for domiciling. The replication system's top priority is to prevent the loss of data and it may override the zone configurations if necessary to ensure data durability. For more information, see [Replication Controls](/docs/v24.3/configure-replication-zones#types-of-constraints).
- If your [log files](/docs/v24.3/logging-overview) are kept in the region where they were generated, there is some cross-region leakage (like the system tables described previously), but the majority of user data that makes it into the logs is going to be homed in that region. If that's not strong enough, you can use the [log redaction functionality](/docs/v24.3/configure-logs#redact-logs) to strip all raw data from the logs. You can also limit your log retention entirely.
- If you start a node with a [`--locality`](/docs/v24.3/cockroach-start#locality) flag that says the node is in region *A*, but the node is actually running in some region *B*, data domiciling based on the inferred node placement will not work. A CockroachDB node only knows its locality based on the text supplied to the `--locality` flag; it can not ensure that it is actually running in that physical location.
