The `SHOW STATEMENTS` [statement](/docs/v24.3/sql-statements) lists details about currently active SQL queries, including:

- The internal ID of the query
- The node executing the query
- The SQL query itself
- How long the query has been running
- The client address, application name, and user that issued the query
- The ID for the current session

These details let you monitor the progress of active queries and, if necessary, identify those that may need to be [cancelled](/docs/v24.3/cancel-query) to prevent unwanted resource consumption.

<Note>
Schema changes and [`BACKUP`](/docs/v24.3/backup)/[`RESTORE`](/docs/v24.3/restore) statements are not executed as queries internally and so are not listed by `SHOW STATEMENTS`. To monitor such statements, use [`SHOW JOBS`](/docs/v24.3/show-jobs) instead.
</Note>

## Aliases

In CockroachDB, the following are aliases for `SHOW STATEMENTS`:

- `SHOW QUERIES`

## Required privileges

All users can see their own currently active queries. Users with the [`VIEWACTIVITY` or `VIEWACTIVITYREDACTED` privilege](/docs/v24.3/security-reference/authorization#supported-privileges) can view see all users' currently active queries. `VIEWACTIVITYREDACTED` causes constants in queries being executed by other users to be redacted.

## Synopsis

<Frame>
  ![SQL syntax diagram](/images/generated/docs/v24.3/show-statements/syntax-diagram-6bdd04b800b5.svg)
</Frame>

- To list the active queries across all nodes of the cluster, use `SHOW STATEMENTS` or `SHOW CLUSTER STATEMENTS`.
- To list the active queries just on the local node, use `SHOW LOCAL STATEMENTS`.

## Response

The following fields are returned for each query:

<table><thead><tr><th>Field</th><th>Description</th></tr></thead><tbody><tr><td><code>query&#95;id</code></td><td>The ID of the query.</td></tr><tr><td><code>node&#95;id</code></td><td>The ID of the node.</td></tr><tr><td><code>session&#95;id</code></td><td>The ID of the session.</td></tr><tr><td><code>user&#95;name</code></td><td>The username of the connected user.</td></tr><tr><td><code>start</code></td><td>The <a href="/docs/v24.3/timestamp"><code>timestamptz</code></a> at which the query started.</td></tr><tr><td><code>query</code></td><td>The SQL query.</td></tr><tr><td><code>client&#95;address</code></td><td>The address and port of the client that issued the SQL query.</td></tr><tr><td><code>application&#95;name</code></td><td>The <a href="/docs/v24.3/set-vars#supported-variables">application name</a> specified by the client, if any. For queries from the <a href="/docs/v24.3/cockroach-sql">built-in SQL client</a>, this will be <code>&#36; cockroach sql</code>.</td></tr><tr><td><code>distributed</code></td><td>If <code>true</code>, the query is being executed by the Distributed SQL (DistSQL) engine. If <code>false</code>, the query is being executed by the standard "local" SQL engine. If <code>NULL</code>, the query is being prepared and it's not yet known which execution engine will be used.</td></tr><tr><td><code>phase</code></td><td>The phase of the query's execution. If <code>preparing</code>, the statement is being parsed and planned. If <code>executing</code>, the statement is being executed.</td></tr></tbody></table>

### Filter for specific queries

You can use a [`SELECT`](/docs/v24.3/select-clause) statement to filter the list of active queries by one or more of the [response fields](#response).

#### Show all queries on node 2

```sql
> WITH x as (SHOW CLUSTER STATEMENTS) SELECT * FROM x
      WHERE node_id = 2;
```

```text
              query_id             | node_id |            session_id            | user_name |              start               |                                 query                                 | client_address  | application_name | distributed |   phase
+----------------------------------+---------+----------------------------------+-----------+----------------------------------+-----------------------------------------------------------------------+-----------------+------------------+-------------+-----------+
  15f92b1cb931f6900000000000000002 |       2 | 15f92b10b92ed4080000000000000002 | root      | 2020-03-04 17:49:06.363515+00:00 | UPSERT INTO vehicle_location_histories VALUES ($1, $2, now(), $3, $4) | 127.0.0.1:65092 |                  |    NULL     | preparing
(1 row)
```

#### Show all queries from a specific address and user

```sql
> WITH x as (SHOW CLUSTER STATEMENTS) SELECT * FROM x
      WHERE client_address = '127.0.0.1:65196' AND user_name = 'maxroach';
```

```text
              query_id             | node_id |            session_id            | user_name |              start               |                     query                     | client_address  | application_name | distributed |   phase
+----------------------------------+---------+----------------------------------+-----------+----------------------------------+-----------------------------------------------+-----------------+------------------+-------------+-----------+
  15f92bf4b27f0b480000000000000002 |       2 | 15f92b7dc85b7ba80000000000000002 | maxroach  | 2020-03-04 18:04:33.964083+00:00 | SELECT city, id FROM vehicles WHERE city = $1 | 127.0.0.1:65196 |                  |    false    | executing
(1 row)
```

### Cancel a query

When you see a query that is taking too long to complete, you can use the [`CANCEL QUERY`](/docs/v24.3/cancel-query) statement to end it.

For example, let's say you use `SHOW CLUSTER STATEMENTS` to find queries that have been running for more than 3 hours:

```sql
> WITH x as (SHOW CLUSTER STATEMENTS) SELECT * FROM x
      WHERE start < (now() - INTERVAL '3 hours');
```

```text
              query_id             | node_id |            session_id            | user_name |              start              |        query        | client_address  | application_name | distributed |   phase
+----------------------------------+---------+----------------------------------+-----------+---------------------------------+---------------------+-----------------+------------------+-------------+-----------+
  15f92c745fe203600000000000000001 |       1 | 15f92c63d4b393b80000000000000001 | root      | 2020-03-04 18:13:42.33385+00:00 | SELECT * FROM rides | 127.0.0.1:65287 | $ cockroach sql  |    true     | executing
(1 row)
```

To cancel this long-running query, and stop it from consuming resources, you note the `query_id` and use it with the `CANCEL QUERY` statement:

```sql
> CANCEL QUERY '15f92c745fe203600000000000000001';
```
